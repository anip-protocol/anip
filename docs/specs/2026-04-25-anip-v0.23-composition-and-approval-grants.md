# ANIP v0.23 Composition And Approval Grants

Date: 2026-04-25

## Context

The GTM question-bank regression exposed a modeling gap:

- The user intent was "show enrichment context for the top at-risk accounts".
- The current GTM estate had separate primitive capabilities for risk selection and account enrichment.
- The agent had to stitch those primitives together, including edge cases such as empty intermediate results.

The immediate fix should not change ANIP core semantics. For the GTM benchmark, the supported business action should be exposed as a normal GTM capability:

`gtm.at_risk_account_enrichment_summary`

That capability can internally select at-risk accounts, enrich them, and return success with an empty account list when no matching at-risk accounts exist.

## v0.23 Composition Direction

ANIP v0.23 should make this pattern explicit at the contract level instead of relying on hidden service implementation details.

Proposed additions:

- `capability.kind`: `atomic` or `composed`.
- `composition.steps`: ordered internal capability/service steps.
- `composition.input_mapping`: how parent inputs feed child steps.
- `composition.output_mapping`: how child outputs become the final bounded response.
- `composition.empty_result_policy`: for example `return_success_no_results`, `clarify`, or `deny`.
- `composition.failure_policy`: how to handle child clarification, denial, approval, or internal errors.
- `composition.audit_policy`: record child invocations under one parent task and preserve lineage.
- `composition.authority_boundary`: `same_service`, `same_package`, or `external_service`.

The agent should still choose one business capability from the service brief. The service or ANIP runtime should own the internal composition.

## Current Approval Behavior

The current GTM approval flow is request-state based, not token based:

- A capability returns `approval_required` with an `approval_request_id`.
- The approval store records a pending request with a preview.
- An approver marks the request `approved`.
- The current code does not mint a continuation token.
- The current code does not enforce one-time consumption of an approved request.
- The regression harness verifies that a request moves from pending to approved; it does not execute a post-approval continuation.

This is acceptable for the showcase, but incomplete for production ANIP.

## v0.23 Approval Grant Direction

ANIP v0.23 should introduce explicit approval grants.

An approval grant should be a signed, auditable authorization object returned after approval. It may be represented as an ANIP token or as a token-bound grant referenced by a token.

Required fields:

- `grant_id`
- `grant_type`: `one_time` or `session_bound`
- `capability`
- `scope`
- `approved_parameters_digest`
- `preview_digest`
- `requester`
- `approver`
- `issued_at`
- `expires_at`
- `max_uses`
- `session_id`, when `grant_type=session_bound`

Recommended semantics:

- `one_time` grants are consumed on first successful mutation or approved action.
- `session_bound` grants are reusable only within a declared session and bounded expiry.
- Grants must be bound to capability, scope, actor lineage, and approved parameter digest.
- Services must reject grants whose parameters drift from the approved preview.
- Audit entries must link the original `approval_request_id`, approval grant, and post-approval invocation.

This keeps approval state out of the agent. The agent presents the approval boundary and then supplies the returned grant on the continuation call.

## Non-Goals For The Current Fix

- Do not change ANIP core schema in the GTM benchmark fix.
- Do not add agent-specific branching for individual regions or empty data cases.
- Do not weaken the question-bank expectation that a valid bounded no-results answer is a successful outcome.
