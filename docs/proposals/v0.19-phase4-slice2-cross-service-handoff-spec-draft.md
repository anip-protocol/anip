# ANIP v0.19 Phase 4 Slice 2: Spec Draft â€” Cross-Service Handoff Hints

## Overview

This document defines a narrow Phase 4 Slice 2 extension for `anip/0.19`. After v0.18 established cross-service continuity (task_id propagation, parent_invocation_id linkage, upstream_service), this slice makes adjacent-service relationships machine-visible.

**Key design decisions:**
- One new optional block on `CapabilityDeclaration`: `cross_service`
- Inside it, three optional arrays of `ServiceCapabilityRef`: `handoff_to`, `refresh_via`, `verify_via`
- `ServiceCapabilityRef` is a simple `{service, capability}` pair â€” advisory, not resolvable
- `ServiceCapabilityRef.service` SHOULD match the value that service uses for `upstream_service` (v0.18) and `metadata.service_name` (discovery) â€” this is a SHOULD, not a MUST, since ANIP cannot enforce cross-service naming consistency
- All hints are advisory only â€” no execution obligations, no workflow semantics
- Same-manifest hints from v0.17 (`refresh_via`, `verify_via`) remain unchanged and separate
- Four relationship types: `handoff_to` (direct output consumer), `refresh_via` (stale-state refresh), `verify_via` (post-action verification), `followup_via` (delayed/async task follow-up)

**What this does NOT add:**
- No workflow graphs, DAGs, or orchestration
- No service discovery or global registry
- No cross-service permission model
- No automatic invocation chaining

**Relationship to v0.17 (same-manifest hints):**

| v0.17 field | Scope | v0.19 cross_service equivalent |
|-------------|-------|-------------------------------|
| `refresh_via` | Same manifest | `cross_service.refresh_via` (cross-service) |
| `verify_via` | Same manifest | `cross_service.verify_via` (cross-service) |
| (none â€” `follows` was rejected) | â€” | `cross_service.handoff_to` (direct output consumer on another service) |
| (none) | â€” | `cross_service.followup_via` (delayed/async task follow-up on another service) |

The same-manifest fields stay separate because same-manifest references are cheap, validated by conformance, and have a different trust model than cross-service references (which are opaque and unvalidated).

## 1. New Type: `ServiceCapabilityRef`

A reference to a capability on an adjacent service. Advisory only â€” ANIP does not validate or resolve it.

```json
{
  "service": "travel-booking",
  "capability": "book_flight"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | Yes | Publisher-supplied identifier for the adjacent service |
| `capability` | string | Yes | Capability name on that service |

### Semantics

- The pair is an **advisory hint**, not a resolvable pointer
- ANIP does not require a global registry to validate or resolve it
- Agents and tools MAY use local knowledge, discovery catalogs, or deployment context to match the reference to a real service
- A referenced service may be unreachable, unavailable, or unauthorized â€” the hint makes no guarantees

### Service identity alignment

`ServiceCapabilityRef.service` SHOULD use the same identifier string that the referenced service uses for:
- `upstream_service` in v0.18 invoke requests (the value agents pass when calling that service)
- `metadata.service_name` in that service's discovery response

This is a SHOULD, not a MUST â€” ANIP cannot enforce cross-service naming consistency. But consistent naming means agents can match `cross_service.handoff_to[].service` directly against `upstream_service` values seen in audit, without maintaining a separate alias map.

## 2. New Field: `CapabilityDeclaration.cross_service`

An optional block containing cross-service relationship hints.

```json
{
  "book_flight": {
    "description": "Book a flight reservation",
    "refresh_via": ["search_flights"],
    "cross_service": {
      "handoff_to": [
        {"service": "payment-service", "capability": "process_payment"}
      ],
      "refresh_via": [
        {"service": "travel-search", "capability": "search_flights"}
      ],
      "verify_via": [
        {"service": "booking-status", "capability": "get_booking_status"}
      ],
      "followup_via": [
        {"service": "fulfillment", "capability": "check_fulfillment_status"}
      ]
    }
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `cross_service` | object | No | Advisory cross-service relationship block |
| `cross_service.handoff_to` | `ServiceCapabilityRef[]` | No | Adjacent-service capabilities that directly consume this capability's output |
| `cross_service.refresh_via` | `ServiceCapabilityRef[]` | No | Adjacent-service capabilities that refresh stale state needed before retry |
| `cross_service.verify_via` | `ServiceCapabilityRef[]` | No | Adjacent-service capabilities that verify this capability's side effects |
| `cross_service.followup_via` | `ServiceCapabilityRef[]` | No | Adjacent-service capabilities for delayed or async task-local follow-up |

### Why `handoff_to` and `followup_via` are separate

Both point to another service, but they express different relationships:
- `handoff_to`: the listed capability directly consumes this capability's output â€” "search produces a quote, booking consumes it"
- `followup_via`: the listed capability provides later status or continuation â€” "order is placed, fulfillment service tracks it"

Merging them would force agents to guess whether a listed capability is the immediate next consumer or a delayed status checker. Keeping them separate preserves semantic clarity.

---

## 3. Advisory Nature

All `cross_service` fields are strictly advisory:

- Services MAY omit the block entirely
- Agents MUST NOT treat the block as an execution mandate
- Services MUST NOT assume referenced services are reachable, available, or authorized
- The block MUST NOT alter existing permission, budget, binding, recovery, or continuity semantics
- The block does NOT change `task_id`, `parent_invocation_id`, or `upstream_service` behavior from v0.18

---

## 4. Normative Guidance

### `handoff_to`

> This capability commonly produces a result that a listed adjacent-service capability can consume in the same logical task.

- SHOULD be used only for direct, adjacent handoffs (one hop)
- SHOULD NOT encode fan-out trees or distant workflow paths
- A listed capability SHOULD be a plausible next consumer of this capability's output

### `cross_service.refresh_via`

> If downstream work fails because upstream state is stale, the listed adjacent-service capability is a likely refresh source.

- SHOULD point to the service that can reissue or refresh the upstream artifact
- SHOULD NOT be used for same-manifest refresh paths (use v0.17 `refresh_via` instead)

### `cross_service.verify_via`

> After this capability causes a side effect, the listed adjacent-service capability is a likely verifier.

- Is post-action guidance, not pre-execution gating
- SHOULD point to a service that is a natural observer or status reader for the effect

### `cross_service.followup_via`

> After this capability completes, later task-local follow-up commonly happens through the listed adjacent-service capability.

- SHOULD be used for delayed or async continuation that lives on another service
- Is NOT the same as `handoff_to` â€” `handoff_to` is for direct output consumers; `followup_via` is for later status checks, polling, or fulfillment tracking
- SHOULD NOT encode full workflow branches or business process diagrams

## 5. Examples

### Travel: search produces quote, booking service consumes it

```json
{
  "search_flights": {
    "cross_service": {
      "handoff_to": [
        {"service": "travel-booking", "capability": "book_flight"}
      ]
    }
  }
}
```

### Travel: stale quote refresh from search service

```json
{
  "book_flight": {
    "cross_service": {
      "refresh_via": [
        {"service": "travel-search", "capability": "search_flights"}
      ]
    }
  }
}
```

### DevOps: deployment verified by monitoring service

```json
{
  "deploy_service": {
    "cross_service": {
      "verify_via": [
        {"service": "monitoring", "capability": "get_deployment_status"}
      ]
    }
  }
}
```

### Order: delayed fulfillment follow-up

```json
{
  "submit_order": {
    "cross_service": {
      "followup_via": [
        {"service": "fulfillment", "capability": "check_fulfillment_status"}
      ]
    }
  }
}
```

## 6. What This Does NOT Introduce

- No remote schema validation
- No typed artifact contracts across services
- No automatic invocation chaining
- No service discovery protocol
- No global capability namespace
- No reciprocal relationship requirements
- No orchestration plans or branching logic
- No changes to v0.17 same-manifest hints
- No changes to v0.18 continuity fields
- No new failure types or recovery classes
