# ANIP v0.15 Slice 2: Spec Draft â€” Authority and Blocked-Action Clarity

## Overview

This document defines concrete protocol additions for v0.15 Slice 2. It covers exact field names, types, validation rules, and service behavior requirements.

**Key design decisions:**
- Authority gaps get a structured `reason_type` vocabulary, not free-form strings
- Blocked-action failures distinguish terminal vs resolvable vs externally-resolvable blocks
- `resolution.action` gets a canonical vocabulary that agents can switch on
- `DeniedCapability` gains structured fields (currently just capability + reason)
- All changes are additive â€” existing fields remain, new fields add precision

## 1. Structured Authority Posture in Permission Discovery

### Problem

`RestrictedCapability.reason` and `DeniedCapability.reason` are free-form strings. Agents must parse human-readable text to understand WHY something is blocked. This creates wrapper logic everywhere.

### Addition: `reason_type` on RestrictedCapability and DeniedCapability

A machine-readable enum classifying the authority gap:

```json
{
  "restricted": [
    {
      "capability": "book_flight",
      "reason": "delegation chain lacks scope: travel.book",
      "reason_type": "insufficient_scope",
      "grantable_by": "human:alice@company.com",
      "resolution_hint": "request_broader_delegation"
    },
    {
      "capability": "execute_trade",
      "reason": "missing cost_ceiling in delegation",
      "reason_type": "unmet_control_requirement",
      "grantable_by": "human:admin@company.com",
      "unmet_token_requirements": ["cost_ceiling"],
      "resolution_hint": "request_budget_bound_delegation"
    }
  ],
  "denied": [
    {
      "capability": "admin_override",
      "reason": "requires admin principal, current chain root is standard user",
      "reason_type": "non_delegable",
      "terminal": true
    },
    {
      "capability": "delete_production",
      "reason": "destructive action requires explicit human authorization",
      "reason_type": "requires_external_approval",
      "terminal": false,
      "escalation_target": "human:ops-lead@company.com"
    }
  ]
}
```

### `reason_type` vocabulary

| Value | Meaning | Typical resolution |
|-------|---------|-------------------|
| `insufficient_scope` | Delegation chain lacks required scope(s) | Request broader delegation from `grantable_by` |
| `insufficient_delegation_depth` | Token is at max delegation depth | Request new root delegation |
| `stronger_delegation_required` | Token needs explicit capability binding | Reissue token with specific capability purpose |
| `unmet_control_requirement` | Token-evaluable control requirement not satisfied | See `unmet_token_requirements` for specifics |
| `non_delegable` | Authority cannot be delegated at all â€” requires direct principal action | Hard stop for agents |
| `requires_external_approval` | Action needs approval from an external authority | Escalate to `escalation_target` |
| `principal_class_insufficient` | Root principal's class (role, tier) is too low | Escalate to higher-class principal |

### New fields on RestrictedCapability

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `reason_type` | string | Yes | One of the `reason_type` vocabulary values |
| `resolution_hint` | string | No | Suggested `resolution.action` value â€” helps agents know what to do without invoking first |

### New fields on DeniedCapability

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `reason_type` | string | Yes | One of the `reason_type` vocabulary values |
| `terminal` | boolean | Yes | If `true`, no agent action can resolve this â€” the authority gap is permanent for this delegation chain |
| `escalation_target` | string | No | Principal identifier who could resolve this (for non-terminal denials) |

## 2. Canonical Resolution Action Vocabulary

### Problem

`resolution.action` is currently a free-form string. Every service invents its own action names. Agents can't switch on them reliably.

### Addition: Canonical vocabulary for `resolution.action`

| Action | Meaning | Agent behavior |
|--------|---------|----------------|
| `retry` | Same request may succeed (transient failure) | Retry with backoff |
| `retry_after_change` | Request may succeed after an external change | Wait for `estimated_availability`, then retry |
| `request_broader_delegation` | Need a token with wider scope | Request from `grantable_by` |
| `request_budget_increase` | Need a token with higher budget | Request from `grantable_by` |
| `request_budget_bound_delegation` | Need a token with budget constraint added | Request from `grantable_by` |
| `request_capability_binding` | Need a token with explicit capability purpose | Request from `grantable_by` |
| `obtain_binding` | Need to invoke a prior capability to get a binding reference | Invoke the source capability |
| `refresh_binding` | Binding is stale, need a fresh one | Re-invoke the source capability |
| `escalate_to_human` | Agent cannot resolve; human intervention needed | Present to `grantable_by` with context |
| `escalate_to_principal` | Need a higher-authority principal to act | Route to `escalation_target` |
| `stop` | Terminal â€” no recovery path exists | Abandon this action path |
| `replan` | Current approach won't work; try a different capability or sequence | Agent replans |

Services SHOULD use these canonical actions. Services MAY use custom action strings for domain-specific recovery, but canonical actions MUST be preferred when applicable.

### Addition: `block_class` on failure responses

A new top-level field on failure objects classifying the block:

```json
{
  "success": false,
  "failure": {
    "type": "insufficient_authority",
    "detail": "delegation chain lacks scope: travel.book",
    "block_class": "resolvable",
    "resolution": {
      "action": "request_broader_delegation",
      "requires": "delegation.scope += travel.book",
      "grantable_by": "human:alice@company.com"
    },
    "retry": true
  }
}
```

| `block_class` | Meaning |
|---------------|---------|
| `transient` | Temporary â€” retry may succeed |
| `resolvable` | Can be fixed by the agent or its delegation chain |
| `externally_resolvable` | Needs action from someone outside the current delegation chain |
| `terminal` | Cannot be resolved â€” permanent block |

## 3. Failure Type Additions

### New failure types

| Type | When | block_class |
|------|------|-------------|
| `insufficient_authority` | Generic authority gap (replaces bare 403) | `resolvable` or `externally_resolvable` |
| `delegation_depth_exceeded` | Token at max delegation depth | `resolvable` (new root delegation) |
| `non_delegable_action` | Action requires direct principal, cannot be delegated | `terminal` |

### Updated existing failure types with `block_class`

Services SHOULD include `block_class` on ALL failure responses. Existing failure types map to:

| Existing type | Default block_class |
|---------------|-------------------|
| `scope_insufficient` | `resolvable` |
| `budget_exceeded` | `resolvable` |
| `budget_currency_mismatch` | `resolvable` |
| `budget_not_enforceable` | `resolvable` |
| `binding_missing` | `resolvable` |
| `binding_stale` | `resolvable` |
| `control_requirement_unsatisfied` | `resolvable` |
| `purpose_mismatch` | `resolvable` |
| `authentication_required` | `resolvable` |
| `invalid_token` | `resolvable` |
| `capability_unavailable` | `transient` |
| `internal_error` | `transient` |

## 4. Permission Discovery and Invoke Consistency

### Principle

The information in `/anip/permissions` and the failure response from `/anip/invoke` should use the same vocabulary. An agent that sees `reason_type: "insufficient_scope"` in permissions should get `resolution.action: "request_broader_delegation"` in the invoke failure.

### Consistency rule

When a capability appears in `restricted` with `resolution_hint: X`, an invoke attempt for that capability SHOULD produce a failure with `resolution.action == X`.

## What this does NOT introduce

- No approval protocol or approval workflow
- No role definitions or role hierarchy
- No IAM integration surface
- No escalation execution (only posture declaration)
- No multi-step recovery orchestration
- No policy language

These are pure authority posture additions: deterministic, stateless at the protocol level, actionable at the agent boundary.

## Scenario acceptance criteria

### Travel: scope restriction with clear recovery

| Step | Expected behavior |
|------|------------------|
| Agent queries permissions with search-only token | `book_flight` in `restricted` with `reason_type: "insufficient_scope"`, `resolution_hint: "request_broader_delegation"` |
| Agent invokes `book_flight` anyway | Failure with `block_class: "resolvable"`, `resolution.action: "request_broader_delegation"`, `grantable_by: human:...` |
| Agent requests broader delegation | Gets new token, booking succeeds |

### DevOps: non-delegable action

| Step | Expected behavior |
|------|------------------|
| Agent queries permissions | `destroy_cluster` in `denied` with `reason_type: "non_delegable"`, `terminal: true` |
| Agent invokes anyway | Failure with `block_class: "terminal"`, `resolution.action: "stop"` |
| Agent knows not to retry or escalate | Clean stop, no wrapper logic needed |

### Finance: external approval required

| Step | Expected behavior |
|------|------------------|
| Agent queries permissions | `execute_large_trade` in `denied` with `reason_type: "requires_external_approval"`, `terminal: false`, `escalation_target: "human:compliance@..."` |
| Agent invokes anyway | Failure with `block_class: "externally_resolvable"`, `resolution.action: "escalate_to_principal"` |
| Agent escalates to the right person | No custom routing logic needed â€” protocol told it who |
