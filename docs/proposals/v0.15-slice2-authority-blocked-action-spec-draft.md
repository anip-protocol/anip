# ANIP v0.15 Slice 2: Spec Draft — Authority and Blocked-Action Clarity

## Overview

This document defines concrete protocol additions for v0.15 Slice 2. It covers exact field names, types, validation rules, and service behavior requirements.

**Key design decisions:**
- Authority gaps get a structured `reason_type` vocabulary — all values are protocol-verifiable, not org-specific
- `denied` remains terminal by definition — resolvable-by-escalation stays under `restricted`
- Resolution.action gets 4 new authority-specific canonical values — binding/retry/replan vocabulary is untouched
- One new failure type: `non_delegable_action` (terminal, hard stop)
- No new cross-cutting taxonomy (`block_class` is NOT added — recoverability is already expressed by existing fields)
- No generic `insufficient_authority` failure — existing specific failure types are preserved

**What this does NOT change:**
- The restricted/denied semantic split (denied = inaccessible to entire chain, always terminal)
- Existing `resolution.action` values from v0.14 (scope, budget, binding, retry vocabulary)
- Existing failure types
- The `retry` field on failures

## 1. Structured Authority Posture in Permission Discovery

### Problem

`RestrictedCapability.reason` and `DeniedCapability.reason` are free-form strings. Agents must parse human-readable text to understand WHY something is blocked.

### Addition: `reason_type` on RestrictedCapability and DeniedCapability

A machine-readable enum classifying the authority gap. All values are things the protocol can verify from the token and capability declaration — no org-specific IAM concepts.

**Permission discovery response with `reason_type`:**

```json
{
  "available": [
    {
      "capability": "search_flights",
      "scope_match": "travel.search"
    }
  ],
  "restricted": [
    {
      "capability": "book_flight",
      "reason": "delegation chain lacks scope: travel.book",
      "reason_type": "insufficient_scope",
      "grantable_by": "human:alice@company.com",
      "resolution_hint": "request_broader_delegation"
    },
    {
      "capability": "execute_trade",
      "reason": "missing cost_ceiling in delegation",
      "reason_type": "unmet_control_requirement",
      "grantable_by": "human:admin@company.com",
      "unmet_token_requirements": ["cost_ceiling"],
      "resolution_hint": "request_budget_bound_delegation"
    },
    {
      "capability": "cancel_booking",
      "reason": "token at max delegation depth",
      "reason_type": "insufficient_delegation_depth",
      "grantable_by": "human:alice@company.com",
      "resolution_hint": "request_broader_delegation"
    }
  ],
  "denied": [
    {
      "capability": "admin_override",
      "reason": "requires admin principal, current chain root is standard user",
      "reason_type": "non_delegable"
    },
    {
      "capability": "destroy_cluster",
      "reason": "destructive action cannot be delegated to agents",
      "reason_type": "non_delegable"
    }
  ]
}
```

### `reason_type` vocabulary

All values are protocol-verifiable from the token and capability declaration:

| Value | Meaning | Where it appears |
|-------|---------|-----------------|
| `insufficient_scope` | Delegation chain lacks required scope(s) | `restricted` |
| `insufficient_delegation_depth` | Token is at max delegation depth | `restricted` |
| `stronger_delegation_required` | Token needs explicit capability binding | `restricted` |
| `unmet_control_requirement` | Token-evaluable control requirement not satisfied (see `unmet_token_requirements`) | `restricted` |
| `non_delegable` | Authority cannot be delegated — requires direct principal action | `denied` |

**Why only 5 values:** These are the authority gaps the protocol can actually identify from structured token data. Org-specific concepts like "role too low" or "needs external approval" are outside the protocol boundary — services express those through the existing `reason` free-text field and `grantable_by`.

### New fields on RestrictedCapability

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `reason_type` | string | Yes | One of the `reason_type` vocabulary values |
| `resolution_hint` | string | No | Suggested `resolution.action` value — helps agents plan recovery without invoking first |

### New field on DeniedCapability

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `reason_type` | string | Yes | Currently always `"non_delegable"` — the only protocol-verifiable reason for full denial |

Note: `denied` capabilities are terminal by definition (SPEC §4.4). No `terminal` field is added — denial already means "inaccessible to the agent's entire delegation chain."

## 2. Authority-Specific Resolution Actions

### Problem

`resolution.action` is currently a free-form string. Services invent their own names. Agents can't reliably switch on them.

### Addition: 4 new canonical values for authority-related recovery

These join the existing vocabulary — they do NOT replace or rename any existing action values.

| Action | Meaning | When to use |
|--------|---------|------------|
| `request_broader_delegation` | Need a token with wider scope or deeper delegation depth | `insufficient_scope`, `insufficient_delegation_depth` |
| `request_capability_binding` | Need a token with explicit capability purpose binding | `stronger_delegation_required` |
| `request_budget_bound_delegation` | Need a token with budget constraint added | `unmet_control_requirement` (cost_ceiling) |
| `stop` | Terminal — no recovery path exists for this delegation chain | `non_delegable` (denied capabilities) |

**Existing actions are unchanged.** Values like `request_scope_grant`, `request_budget_increase`, `obtain_binding`, `refresh_binding`, `wait_and_retry`, `request_new_delegation`, `obtain_quote_first`, and `request_matching_currency_delegation` remain valid. Services MAY use any action string — canonical values are RECOMMENDED, not required.

### Migration note

`request_scope_grant` and `request_broader_delegation` express similar intent. Services SHOULD prefer `request_broader_delegation` for new implementations but MAY continue using `request_scope_grant`. No deprecation — both are valid.

## 3. New Failure Type: `non_delegable_action`

### When

The invoked capability requires direct principal action and cannot be performed through any delegation chain. This is a hard stop — no amount of re-delegation, scope widening, or escalation will fix it.

```json
{
  "success": false,
  "failure": {
    "type": "non_delegable_action",
    "detail": "destroy_cluster requires direct admin principal action and cannot be delegated",
    "resolution": {
      "action": "stop",
      "requires": "direct principal invocation, not delegation"
    },
    "retry": false
  }
}
```

How a service signals non-delegable: the service checks whether the capability is marked as non-delegable (service-internal logic — this is NOT a protocol-level declaration, just a failure type). When the service determines the action cannot proceed via delegation, it returns this failure type.

## 4. Permission Discovery and Invoke Consistency

### Principle

The information in `/anip/permissions` and the failure response from `/anip/invoke` should use compatible vocabulary. An agent that sees `resolution_hint: "request_broader_delegation"` in permissions should get `resolution.action: "request_broader_delegation"` when invoking.

### Consistency rule

When a capability appears in `restricted` with `resolution_hint: X`, an invoke attempt SHOULD produce a failure with `resolution.action == X`. This is a SHOULD, not MUST — the invoke path may have more context and choose a more specific action.

---

## What this does NOT introduce

- No `block_class` taxonomy — recoverability is already expressed by `retry`, the action, and whether the capability is restricted vs denied
- No generic `insufficient_authority` failure type — existing specific types are better
- No `terminal` field on failures — denied = terminal by definition; restricted failures have `retry`
- No `escalation_target` — org-specific routing stays outside the protocol
- No `principal_class_insufficient` reason type — not protocol-verifiable
- No `requires_external_approval` reason type — too IAM-adjacent
- No approval protocol or workflow
- No role hierarchy or IAM integration
- No changes to binding/retry/replan vocabulary

## Scenario acceptance criteria

### Travel: scope restriction with clear recovery

| Step | Expected behavior |
|------|------------------|
| Agent queries permissions with search-only token | `book_flight` in `restricted` with `reason_type: "insufficient_scope"`, `resolution_hint: "request_broader_delegation"`, `grantable_by: "human:..."` |
| Agent invokes `book_flight` anyway | Failure with `type: "scope_insufficient"`, `resolution.action: "request_broader_delegation"`, `grantable_by: "human:..."` |
| Agent requests broader delegation from grantable_by | Gets new token with travel.book scope, booking succeeds |

### DevOps: non-delegable action

| Step | Expected behavior |
|------|------------------|
| Agent queries permissions | `destroy_cluster` in `denied` with `reason_type: "non_delegable"` |
| Agent invokes anyway | Failure with `type: "non_delegable_action"`, `resolution.action: "stop"`, `retry: false` |
| Agent stops cleanly | No retry, no escalation attempt — protocol told it this is terminal |

### Finance: control requirement gap

| Step | Expected behavior |
|------|------------------|
| Agent queries permissions | `execute_trade` in `restricted` with `reason_type: "unmet_control_requirement"`, `unmet_token_requirements: ["cost_ceiling"]`, `resolution_hint: "request_budget_bound_delegation"` |
| Agent invokes anyway | Failure with `type: "control_requirement_unsatisfied"`, `resolution.action: "request_budget_bound_delegation"` |
| Agent requests budget-bound delegation | Gets new token with budget, trade succeeds |
