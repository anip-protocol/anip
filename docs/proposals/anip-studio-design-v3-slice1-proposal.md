# ANIP Studio: Design V3 Slice 1 — Design Proposal

## 1. Goal

Establish the authoring foundation for Studio: Design — make requirements, scenarios, and proposal surface declarations safely editable. Replace the evaluator's text-matching heuristic with structured `declared_surfaces` on proposals.

**What this slice delivers:**
- `declared_surfaces` on proposal schema (structured, machine-readable)
- Evaluator prefers `declared_surfaces` over text inference
- Editable requirements + scenario models in Studio (read → draft → export)
- YAML import/export with schema validation
- Dirty-state tracking + basic diff visibility
- Minimal advanced editing surface (form-based, not raw YAML)

**What this slice does NOT deliver:**
- Guided business-friendly UX (Slice 2)
- Scenario builder for non-technical users (Slice 3)
- Agent-assisted generation (Slice 4)
- Persistence / workspaces / collaboration

---

## 2. Proposal Schema: `declared_surfaces`

### Addition

Add optional `declared_surfaces` to `proposal` in `tooling/schemas/proposal.schema.json`:

```yaml
proposal:
  declared_surfaces:
    budget_enforcement: true
    binding_requirements: true
    authority_posture: true
    recovery_class: true
    refresh_via: true
    verify_via: true
    followup_via: false
    cross_service_handoff: false
    cross_service_continuity: true
    cross_service_reconstruction: true
```

### Schema Shape

```json
"declared_surfaces": {
  "type": "object",
  "description": "Machine-readable declaration of which ANIP semantic surfaces are intentionally present in this design. When present, the evaluator prefers this over heuristic text inference.",
  "additionalProperties": false,
  "properties": {
    "budget_enforcement": { "type": "boolean", "default": false },
    "binding_requirements": { "type": "boolean", "default": false },
    "authority_posture": { "type": "boolean", "default": false },
    "recovery_class": { "type": "boolean", "default": false },
    "refresh_via": { "type": "boolean", "default": false },
    "verify_via": { "type": "boolean", "default": false },
    "followup_via": { "type": "boolean", "default": false },
    "cross_service_handoff": { "type": "boolean", "default": false },
    "cross_service_continuity": { "type": "boolean", "default": false },
    "cross_service_reconstruction": { "type": "boolean", "default": false }
  }
}
```

### Evaluator Behavior

When `proposal.declared_surfaces` exists:
- The evaluator reads booleans directly — no text scanning
- `true` = surface is intentionally present → eligible for handled credit
- `false` or absent = surface not declared → glue noted, no credit

When `proposal.declared_surfaces` is absent:
- Fall back to V2 heuristic text matching (backward compat)

### Migration

All 13 existing example packs get `declared_surfaces` added to their proposals, reflecting what's actually in each proposal's text/components. This is a one-time migration, not an ongoing dual path.

## 3. Studio Editing Model

### State Machine

Every artifact in Studio has one of three states:

```
┌──────────┐     edit      ┌──────────┐    export    ┌──────────┐
│   Read   │ ──────────> │  Draft   │ ──────────> │ Exported │
│ (clean)  │ <────────── │ (dirty)  │             │ (valid)  │
└──────────┘    discard    └──────────┘             └──────────┘
```

- **Read** — artifact loaded from pack, no edits. This is the V1/V2 state.
- **Draft** — in-memory edits exist. Artifact is typed + schema-validated on change. Dirty indicator visible.
- **Exported** — user explicitly exports/saves. Output is canonical YAML, schema-valid.

### What's Editable in Slice 1

| Artifact | Editable | How |
|----------|----------|-----|
| Requirements | Yes | Form-based section editing (system, transports, trust, auth, audit, lineage, risk, constraints, scale) |
| Scenario | Yes | Form-based editing (name, category, narrative, context key-values, expected behavior list, expected ANIP support list) |
| Proposal | Partial | Only `declared_surfaces` toggles editable. Rest read-only. Full proposal editing deferred. |
| Evaluation | No | Read-only. Generated by evaluator. |

### Validation

Edits are validated against truth-layer schemas on every change (debounced). The UI shows:
- Green checkmark: valid
- Red indicator + error messages: invalid (which field, why)

This validation uses the same `jsonschema` validation the evaluator uses, ported to TypeScript via `ajv` (the standard JSON Schema validator for JS/TS).

## 4. Import / Export

### Import

- Load from existing pack (current behavior, unchanged)
- Import from YAML file (new: file picker → parse YAML → validate against schema → load into store)

### Export

- Export as YAML (new: serialize current draft → download as `.yaml` file)
- Copy as YAML (new: serialize → copy to clipboard)

Both import and export preserve the canonical artifact shape. No UI-only fields leak into the exported artifact.

## 5. Diff Visibility

### What Changed

When an artifact is in Draft state, the UI shows a simple diff indicator:

- **Field-level change markers** — sections that differ from the original loaded artifact are highlighted with a change badge
- **"Changes" panel** — collapsible panel listing changed fields with before/after values

This is not a full rich diff viewer. It's structured field comparison:

```
── Changes from original ──
system.name: "travel-service" → "travel-booking-service"
transports.grpc: false → true
risk_profile.capabilities.book_flight.cost_visibility_required: false → true
```

Enough for technical trust and handoff review. Rich visual diff deferred.

## 6. Studio UI Changes

### Requirements View → Requirements Editor

The current `RequirementsView.vue` renders read-only sections. In Slice 1, each section becomes form-editable:

- **System** — text inputs for name, domain, deployment intent
- **Transports** — toggle switches for HTTP, stdio, gRPC
- **Trust & Auth** — toggles + dropdowns
- **Permissions** — toggles
- **Audit & Lineage** — toggles
- **Risk Profile** — per-capability cards with editable side_effect, cost_visibility, recovery_guidance
- **Business Constraints** — toggle list
- **Scale** — dropdown + toggle

Each section has an "Edit" / "Done" toggle. When editing, fields become interactive. When done, they collapse back to the read display.

### Scenario Detail → Scenario Editor

The current `ScenarioDetailView.vue` renders read-only narrative + context. In Slice 1:

- **Name** — text input (validated against `^[a-z0-9_\-]+$`)
- **Category** — dropdown (safety, orchestration, observability)
- **Narrative** — textarea
- **Context** — dynamic key-value editor (add/remove/edit keys, values can be string/number/boolean/array)
- **Expected Behavior** — editable string list (add/remove/reorder)
- **Expected ANIP Support** — editable string list

### Proposal View → Declared Surfaces Editor

The current `ProposalView.vue` renders read-only components. In Slice 1, add a `Declared Surfaces` section at the top:

- 10 toggle switches, one per surface
- Each shows the surface name + a short description
- Toggling updates the in-memory proposal
- Rest of proposal remains read-only

### Toolbar

Add a persistent toolbar to all editor views:

```
[Dirty indicator: "3 changes"] [Validate ✓] [Export YAML ↓] [Discard changes ↺]
```

- Dirty indicator: count of changed fields
- Validate: shows validation status (auto-runs on change)
- Export: downloads current state as YAML
- Discard: resets to original loaded state

## 7. Data Architecture

### Editable Store

Extend `studio/src/design/store.ts`:

```typescript
interface DesignState {
  // V1/V2 (unchanged)
  packs: DesignPack[]
  activePackId: string | null
  // V2 (unchanged)
  liveEvaluation: Evaluation | null
  validating: boolean
  validationError: string | null
  apiAvailable: boolean
  // V3 Slice 1 (new)
  draftRequirements: Requirements | null   // in-memory edits
  draftScenario: Scenario | null           // in-memory edits
  draftDeclaredSurfaces: DeclaredSurfaces | null  // proposal surface toggles
  originalRequirements: Requirements | null // for diff
  originalScenario: Scenario | null        // for diff
  isDirty: boolean
  validationErrors: ValidationError[]
}
```

### Schema Validation in Browser

Add `ajv` as a dependency. Load the 3 truth-layer schemas at startup. Validate on every edit (debounced 300ms).

```typescript
import Ajv from 'ajv'
const ajv = new Ajv()
const validateRequirements = ajv.compile(requirementsSchema)
const validateScenario = ajv.compile(scenarioSchema)
```

### YAML Serialization

Add `js-yaml` as a runtime dependency (currently dev-only for the build script). Use it for import (parse) and export (dump).

## 10. V3 Slice 1 Scope vs Deferred

### Slice 1 (this build)

- `declared_surfaces` on proposal schema
- Evaluator prefers structured declarations
- All 13 packs migrated with `declared_surfaces`
- Editable requirements (form sections)
- Editable scenario (form fields)
- Editable declared_surfaces (toggle grid)
- Schema validation (ajv in browser)
- YAML import/export
- Dirty-state tracking
- Basic field-level diff
- Editor toolbar

### Deferred

- Guided business-friendly requirements wizard (Slice 2)
- Guided scenario builder for non-technical users (Slice 3)
- Agent-assisted generation + critique (Slice 4)
- Full proposal editing (beyond declared_surfaces)
- Persistence / workspaces
- Collaboration / commenting
- Rich visual diff viewer

---

## 11. Success Criteria

- Requirements round-trip: load → edit → export → re-load → identical structure
- Scenario round-trip: same guarantee
- `declared_surfaces` on proposals: evaluator uses them, scores change when toggles change
- No schema-invalid state can be exported
- Diff shows what changed from original
- Old proposals without `declared_surfaces` still work (fallback to V2 heuristic)
