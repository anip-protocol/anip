# ANIP v0.13 Slice 1 Proposal: Budget, Binding, And Constraint Enforcement

## Why This Slice Comes First

The strongest current `Execution Scenario Validation` results still land on
`PARTIAL` for a simple reason:

- the interface can expose constraints
- but it cannot yet enforce enough of them at the execution boundary

This is most visible in the travel scenarios:

- single-service booking under a hard budget
- multi-service search then book with a bounded offer or quote

Those cases do not primarily need more lineage, more audit, or more recovery
language.

They need stronger pre-execution control.

That is why `v0.13` should start with Slice 1.

## Slice 1 Goal

Make ANIP better at expressing and enforcing the conditions under which an
action is allowed to execute.

In practical terms, this slice should let a service say:

- this action must stay under this cost ceiling
- this action is only valid when bound to this prior quote or offer
- this action is only valid within this freshness window
- this action must be rejected if these control requirements are not satisfied

Without introducing:

- workflow sequencing
- a policy language
- an approval engine
- a business-rules platform

## The Core Problem

Today ANIP already has useful surfaces for:

- estimated cost
- permission discovery
- structured failures
- blocked-action hints

But there is still a gap between:

- visible constraint

and:

- enforceable control

That gap is exactly where teams keep writing:

- budget wrapper logic
- bound-offer validation code
- freshness checks around execution
- service-to-service guard glue

## Slice 1 Scope

Slice 1 should focus on three tightly related additions:

1. enforceable cost and budget constraints
2. execution-time binding requirements
3. explicit control requirements

These three belong together because they all answer the same question:

> what must be true before this action is allowed to execute?

## 1. Enforceable Cost And Budget Constraints

### Intent

Move cost semantics from informative-only toward execution-relevant control.

### Candidate Direction

- clarify how delegation or request-level constraints can carry a cost ceiling
- clarify how services compare declared or computed cost against that ceiling
- require deterministic rejection behavior when cost exceeds the bound
- preserve distinction between:
  - fixed cost
  - bounded estimate
  - dynamic unresolved cost

### Expected Impact

- less budget-wrapper glue
- less prompt-level "stay under X" logic
- less external preflight policy code for simple bounded actions

## 2. Execution-Time Binding Requirements

### Intent

Let one step produce a control artifact that a later step must be bound to at
execution time.

Typical examples:

- quote ID
- offer reference
- price lock reference
- validity token

### Candidate Direction

- declare that a capability requires a bound reference of a known class
- let invocation carry the bound reference in a machine-readable way
- require rejection when the reference is missing, stale, or invalid
- keep the semantics about binding, not workflow ordering

### Expected Impact

- less custom glue between planning output and execution input
- less ad hoc service-to-service validation code
- cleaner multi-service scenario handling

## 3. Explicit Control Requirements

### Intent

Make it possible to distinguish:

- informational prerequisites

from:

- enforceable execution controls

### Candidate Direction

- introduce a small, deterministic control-requirement concept
- support requirement types such as:
  - `cost_ceiling`
  - `bound_reference`
  - `freshness_window`
  - `stronger_delegation_required`
- clarify service behavior when a required control is not satisfied
- ensure failure output remains structured and scenario-evaluable

### Expected Impact

- less "only invoke if ..." wrapper logic
- fewer hidden execution assumptions
- more portable behavior across runtimes

## Proposed Success Criteria

Slice 1 succeeds if scenario results move in the following direction:

- travel single-service budget scenario:
  - from `PARTIAL`
  - toward `HANDLED`
- travel multi-service budget-and-binding scenario:
  - from `PARTIAL`
  - toward `HANDLED`

And the remaining glue should narrow to:

- organization-specific approval routing
- organization-specific policy choices

Not:

- basic budget enforcement
- basic binding checks
- basic execution-time control validation

## What This Slice Should Not Try To Do

This slice should not introduce:

- approval workflows
- general policy expression
- multi-step orchestration semantics
- generalized dependency graphs
- long-running execution state machines

If a proposal starts needing those, it is no longer a good Slice 1 candidate.

## Validation Standard

Slice 1 should be judged through `Execution Scenario Validation`, not by spec
elegance.

The key question is:

> does this reduce the amount of glue a team must write at the execution
> boundary?

If not, it is probably the wrong `v0.13` move.

## Suggested Next Step

Turn this slice into a narrower spec-diff note covering:

1. constraint representation
2. binding representation
3. failure behavior when controls are missing or violated
4. scenario acceptance tests for the travel examples

That would be the cleanest path from Phase 2 framing to actual spec work.
