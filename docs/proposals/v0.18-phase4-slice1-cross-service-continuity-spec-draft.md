# ANIP v0.18 Phase 4 Slice 1: Spec Draft — Cross-Service Continuity

## Overview

This document defines normative guidance and one new field for v0.18 Phase 4 Slice 1: cross-service continuity and linkage. The goal is to make multi-service execution chains legible and reconstructable.

**Key design decisions:**
- `task_id` and `parent_invocation_id` already exist — this slice adds normative cross-service propagation rules for them
- One new optional field: `upstream_service` on invocation requests — an unvalidated hint identifying which service produced the `parent_invocation_id`
- `client_reference_id` is out of scope — it already has clear semantics (caller-supplied opaque string)
- No workflow graphs, no global registries, no cross-service capability references
- Each service's audit log stays independent — cross-service reconstruction uses shared field values, not a central store

**What this adds:**
- Normative rules for when agents SHOULD propagate `task_id` across services
- Normative rules for when agents SHOULD set `parent_invocation_id` to an upstream service's `invocation_id`
- One new optional field: `upstream_service` — a reconstruction hint (unvalidated, may be wrong or absent)
- Reconstruction guidance (not guarantee): operators can likely reconstruct cross-service chains using `task_id` queries + `parent_invocation_id` links + `upstream_service` hints, but completeness depends on agent cooperation
- Examples for direct handoff, async follow-up, and multi-service fan-out

---

## 1. Cross-Service `task_id` Propagation

### Problem

`task_id` exists but the spec doesn't say what happens across services. An agent calling Service A then Service B with the same `task_id` works — but it's not normative. Teams invent local conventions.

### Normative rule

**When downstream work is part of the same logical task, agents SHOULD propagate the same `task_id` to downstream services.**

Specifically:
- If an agent invokes Service A with `task_id: "trip-2026"`, and then invokes Service B as part of the same task, it SHOULD pass `task_id: "trip-2026"` to Service B
- If the downstream work is a new, independent task (not caused by the upstream), the agent SHOULD use a different `task_id` or omit it
- Services MUST NOT reject a `task_id` simply because it was originally created by a different service — `task_id` is a caller-supplied correlation value, not a service-scoped identifier

### Token purpose interaction

If the delegation token carries `purpose.task_id`, that takes precedence (existing rule). For cross-service scenarios, the orchestrating agent should issue tokens with the same `purpose.task_id` across services to enforce task-level continuity at the token layer.

## 2. Cross-Service `parent_invocation_id` Linkage

### Problem

`parent_invocation_id` exists but the spec doesn't say how it works across services. When Service A returns `invocation_id: "inv-abc123"` and the agent calls Service B, should the agent pass `parent_invocation_id: "inv-abc123"`? Currently ambiguous.

### Normative rule

**When a downstream invocation is directly triggered by an upstream invocation, agents SHOULD set `parent_invocation_id` to the upstream `invocation_id`, even if the upstream was on a different service.**

Specifically:
- If Service A's `search_flights` returns `invocation_id: "inv-abc123"`, and the agent immediately calls Service B's `book_hotel` as a follow-up, it SHOULD pass `parent_invocation_id: "inv-abc123"` to Service B
- If the downstream invocation is not directly caused by a specific upstream invocation (e.g., a scheduled job, a user-initiated action), the agent SHOULD NOT set `parent_invocation_id`
- Services MUST NOT reject a `parent_invocation_id` that was issued by a different service — it is an opaque reference, not a resolvable pointer

### What `parent_invocation_id` means across services

`parent_invocation_id` across services is a **causal link**, not a resolvable reference. The downstream service cannot look up the upstream invocation — it only records the linkage. Reconstruction requires querying both services' audit logs.

## 3. New Field: `upstream_service`

### Problem

When reconstructing a cross-service chain, operators have `task_id` and `parent_invocation_id` to find related entries. But they don't know WHERE the parent invocation lives — which service issued `inv-abc123`? Today they search all known services.

### Addition

A new optional field on invocation requests: `upstream_service`.

```yaml
# Invocation request to Service B
parameters: { hotel: "Hilton SEA", dates: "2026-03-10/2026-03-12" }
task_id: "trip-2026"
parent_invocation_id: "inv-abc123"
upstream_service: "travel-service"
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `upstream_service` | string | No | Identifier of the service that produced the `parent_invocation_id`. Agent-supplied, not validated by the receiving service. |

### Semantics

- `upstream_service` is an unvalidated hint for reconstruction — it suggests where the parent invocation likely lives, but may be wrong, stale, or absent
- The receiving service MUST echo it in the response and SHOULD include it in the audit entry
- The receiving service MUST NOT validate or resolve it — it is an opaque string supplied by the caller
- If `parent_invocation_id` is absent, `upstream_service` SHOULD also be absent

### Why not a service-qualified `parent_invocation_id`?

Embedding the service name in the invocation ID (e.g., `travel-service:inv-abc123`) would change the `inv-{hex12}` format that's already established. A separate field keeps the format stable and makes the hint optional.

## 4. Reconstruction Guidance

### Statement

An operator SHOULD be able to attempt cross-service chain reconstruction using:
1. The shared `task_id` — query each known service's audit log with `?task_id=X`
2. The `parent_invocation_id` links — follow the chain from child to parent
3. The `upstream_service` hints — a suggestion of where to look for each parent (not authoritative)

No central audit store or cross-service query API is required. Each service's audit log is independent. The shared field values provide the linkage.

### Limitations

This is best-effort reconstruction, not a guarantee:
- **Completeness** — if a service was not called with `task_id`, its entries won't appear in the task-scoped query
- **`upstream_service` accuracy** — the field is agent-supplied and unvalidated; it may be wrong, missing, or stale. Operators should treat it as a likely starting point, not an authoritative answer
- **Ordering** — audit timestamps may drift across services; `parent_invocation_id` provides causal ordering, not wall-clock ordering
- **Availability** — a service's audit log may be unavailable at reconstruction time

## 5. Examples

### Direct handoff: search on Service A, book on Service B

```
Agent → Service A (search_flights)
  task_id: "trip-2026"
  → invocation_id: "inv-111aaa"

Agent → Service B (book_hotel)
  task_id: "trip-2026"
  parent_invocation_id: "inv-111aaa"
  upstream_service: "travel-service"
  → invocation_id: "inv-222bbb"
```

Reconstruction: query `travel-service` and `hotel-service` with `?task_id=trip-2026`. The `parent_invocation_id` links `inv-222bbb → inv-111aaa`. The `upstream_service` hint suggests `inv-111aaa` likely lives on `travel-service`.

### Fan-out: one trigger, multiple downstream services

```
Agent → Service A (plan_trip)
  task_id: "trip-2026"
  → invocation_id: "inv-000"

Agent → Service B (book_flight)
  task_id: "trip-2026"
  parent_invocation_id: "inv-000"
  upstream_service: "planning-service"

Agent → Service C (book_hotel)
  task_id: "trip-2026"
  parent_invocation_id: "inv-000"
  upstream_service: "planning-service"

Agent → Service D (book_car)
  task_id: "trip-2026"
  parent_invocation_id: "inv-000"
  upstream_service: "planning-service"
```

All four services share `task_id: "trip-2026"`. Services B, C, D all point to `inv-000` as parent. Reconstruction shows a fan-out from the planning step.

### Async follow-up: delayed downstream

```
Agent → Service A (submit_order)
  task_id: "order-789"
  → invocation_id: "inv-aaa"

  [... time passes ...]

Agent → Service B (check_fulfillment)
  task_id: "order-789"
  parent_invocation_id: "inv-aaa"
  upstream_service: "order-service"
```

Even with a delay, the `task_id` and `parent_invocation_id` preserve the causal link.

## 6. What This Does NOT Introduce

- No cross-service capability references (`service:capability` syntax)
- No cross-service `refresh_via` or `verify_via` (those stay same-manifest)
- No global audit query API
- No central task registry
- No workflow execution semantics
- No service discovery protocol
- No new failure types
- No changes to existing field semantics (`task_id` and `parent_invocation_id` keep their current meaning — this slice adds cross-service propagation guidance)
- `client_reference_id` is not in scope — it already has clear semantics and does not need cross-service rules

## 7. Scenario Acceptance Criteria

### Travel: cross-service trip planning

| Step | Expected |
|------|----------|
| Agent searches flights on Travel Service | `task_id: "trip-2026"` in request and audit |
| Agent books hotel on Hotel Service | Same `task_id: "trip-2026"`, `parent_invocation_id` pointing to search, `upstream_service: "travel-service"` |
| Operator queries both services with `?task_id=trip-2026` | Gets complete cross-service chain |
| Operator follows `parent_invocation_id` + `upstream_service` | Has a hint about where each parent likely lives |

### DevOps: deployment with post-deploy verification on different service

| Step | Expected |
|------|----------|
| Agent deploys on Deploy Service | `task_id: "deploy-42"`, gets `invocation_id: "inv-deploy"` |
| Agent verifies on Monitoring Service | `task_id: "deploy-42"`, `parent_invocation_id: "inv-deploy"`, `upstream_service: "deploy-service"` |
| Operator reconstructs | Both audit logs linked by task_id + parent chain |
