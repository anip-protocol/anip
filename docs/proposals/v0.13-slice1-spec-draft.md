# ANIP v0.13 Slice 1: Spec Draft — Budget, Binding, and Control Requirements

## Overview

This document defines the concrete protocol additions for v0.13 Slice 1. It covers exact field names, validation rules, failure types, and service behavior requirements.

## 1. Enforceable Cost Constraints

### Delegation constraint: `budget`

A delegation token MAY carry a `budget` constraint in its purpose or constraints block. When present, the service MUST enforce it.

**Token issuance request:**

```json
{
  "scope": ["travel.book"],
  "subject": "agent:trip-planner",
  "capability": "book_flight",
  "budget": {
    "currency": "USD",
    "max_amount": 500
  }
}
```

**Token claims (JWT):**

```json
{
  "budget": {
    "currency": "USD",
    "max_amount": 500
  }
}
```

### Service enforcement

When a delegation token carries a `budget` constraint:

1. **Before execution**: If the capability declares a cost and the declared minimum (`cost.financial.range_min`) already exceeds `budget.max_amount`, the service MUST reject with `budget_exceeded` failure **before** calling the handler.

2. **After execution**: If the handler returns `cost_actual` and it exceeds `budget.max_amount`, the service MUST still return the result (the action already happened) but MUST include a `budget_warning` field in the response.

3. **Currency mismatch**: If the token's `budget.currency` does not match the capability's `cost.financial.currency`, the service MUST reject with `budget_currency_mismatch`.

### Budget enforcement rule

```
IF token.budget exists:
  IF capability.cost.financial.range_min > token.budget.max_amount:
    REJECT with budget_exceeded (before execution)
  ELSE:
    EXECUTE handler
    IF cost_actual.amount > token.budget.max_amount:
      INCLUDE budget_warning in response
```

### New failure type: `budget_exceeded`

```json
{
  "success": false,
  "failure": {
    "type": "budget_exceeded",
    "detail": "Capability minimum cost $200 exceeds delegated budget of $150",
    "resolution": {
      "action": "request_budget_increase",
      "requires": "higher_budget_delegation",
      "grantable_by": "human:manager@company.com"
    },
    "retry": false
  },
  "invocation_id": "inv-...",
  "budget_context": {
    "budget_max": 150,
    "budget_currency": "USD",
    "capability_cost_min": 200,
    "capability_cost_max": 800
  }
}
```

### Budget narrowing in delegation chains

Budget constraints MUST narrow, never widen:

- If a parent token has `budget.max_amount = 500`, a child token MUST NOT have `budget.max_amount > 500`
- The service MUST reject token issuance that would widen a budget constraint

This matches the existing scope-narrowing rule.

## 2. Execution-Time Binding Requirements

### Capability declaration: `requires_binding`

A capability MAY declare that it requires a bound reference from a prior step.

**Manifest declaration:**

```json
{
  "book_flight": {
    "description": "Book a flight reservation",
    "side_effect": { "type": "irreversible" },
    "requires_binding": [
      {
        "type": "quote",
        "source_capability": "search_flights",
        "field": "quote_id",
        "max_age": "PT15M"
      }
    ]
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | Yes | Binding type identifier (e.g., `quote`, `offer`, `price_lock`) |
| `source_capability` | string | No | Which capability produces this binding |
| `field` | string | Yes | Which field in the invocation params must carry the reference |
| `max_age` | string (ISO 8601 duration) | No | Maximum age before the binding is considered stale |

### Invocation with binding

The caller MUST include the bound reference in the invocation parameters:

```json
{
  "parameters": {
    "flight_number": "AA100",
    "quote_id": "qt-a1b2c3d4"
  }
}
```

### Service enforcement

When a capability declares `requires_binding`:

1. **Missing binding**: If the required field is absent from parameters, the service MUST reject with `binding_missing`.
2. **Stale binding**: If `max_age` is declared and the binding is older than the threshold, the service MUST reject with `binding_stale`.
3. **Invalid binding**: If the reference doesn't resolve to a valid prior result, the service MUST reject with `binding_invalid`.

### New failure types

**`binding_missing`:**
```json
{
  "type": "binding_missing",
  "detail": "Capability book_flight requires a bound 'quote' reference in field 'quote_id'",
  "resolution": {
    "action": "obtain_binding",
    "requires": "invoke search_flights first to obtain a quote_id"
  },
  "retry": false
}
```

**`binding_stale`:**
```json
{
  "type": "binding_stale",
  "detail": "Quote qt-a1b2c3d4 has expired (max_age: 15 minutes)",
  "resolution": {
    "action": "refresh_binding",
    "requires": "invoke search_flights again to obtain a fresh quote_id"
  },
  "retry": false
}
```

**`binding_invalid`:**
```json
{
  "type": "binding_invalid",
  "detail": "Quote qt-a1b2c3d4 does not exist or has been consumed",
  "resolution": {
    "action": "obtain_new_binding",
    "requires": "invoke search_flights to obtain a valid quote_id"
  },
  "retry": false
}
```

## 3. Explicit Control Requirements

### Capability declaration: `control_requirements`

A capability MAY declare explicit control requirements that must be satisfied at execution time.

**Manifest declaration:**

```json
{
  "execute_trade": {
    "description": "Execute a market trade",
    "side_effect": { "type": "irreversible" },
    "cost": { "certainty": "variable" },
    "control_requirements": [
      { "type": "cost_ceiling", "enforcement": "reject" },
      { "type": "bound_reference", "field": "market_quote_id", "enforcement": "reject" },
      { "type": "freshness_window", "max_age": "PT5M", "enforcement": "reject" }
    ]
  }
}
```

### Control requirement types

| Type | Meaning | What the service checks |
|------|---------|------------------------|
| `cost_ceiling` | Token must carry a budget constraint | `token.budget` must exist |
| `bound_reference` | A prior-step reference must be present | Named field must be in params |
| `freshness_window` | The invocation must happen within a time limit | Binding or invocation must be recent enough |
| `stronger_delegation_required` | A more specific delegation is needed | Token must have explicit capability binding |

### Enforcement values

| Value | Service behavior |
|-------|-----------------|
| `reject` | MUST reject if not satisfied |
| `warn` | SHOULD include a warning but MAY proceed |

### New failure type: `control_requirement_unsatisfied`

```json
{
  "type": "control_requirement_unsatisfied",
  "detail": "Capability execute_trade requires a cost_ceiling in the delegation token",
  "resolution": {
    "action": "request_budget_bound_delegation",
    "requires": "delegation token with budget constraint"
  },
  "retry": false,
  "unsatisfied_requirements": ["cost_ceiling"]
}
```

## Permission discovery extension

`POST /anip/permissions` response SHOULD include control requirement status per capability:

```json
{
  "available": [
    {
      "capability": "search_flights",
      "scope_match": "travel.search",
      "control_requirements_met": true
    }
  ],
  "restricted": [
    {
      "capability": "book_flight",
      "reason": "missing cost_ceiling in delegation",
      "grantable_by": "human:admin@company.com",
      "unmet_requirements": ["cost_ceiling", "bound_reference"]
    }
  ]
}
```

## Transport parity

All new fields are transport-neutral:

| Surface | Transport handling |
|---------|-------------------|
| `budget` in token | Encoded in JWT claims (all transports) |
| `requires_binding` in manifest | Part of capability declaration (all transports) |
| `control_requirements` in manifest | Part of capability declaration (all transports) |
| Binding fields in invoke | In the invocation request body/params (all transports) |
| New failure types | In the failure response body (all transports) |

---

## What this does NOT introduce

- No workflow ordering or sequencing
- No approval protocol
- No policy language
- No generalized dependency graph
- No long-running state

These are pure pre-execution control checks: deterministic, stateless, enforceable at the invoke boundary.

---

## Decision test

1. **Does it help the agent know more before execution?** — Yes (budget limits, binding requirements)
2. **Does it let the service reject earlier and more clearly?** — Yes (3 new deterministic rejection types)
3. **Does it remove repeated safety wrapper code?** — Yes (budget wrappers, binding validation, freshness checks)
4. **Would it still make sense with separate policy systems?** — Yes (these are execution-boundary checks, not policy)
5. **Does it preserve ANIP as execution/control protocol?** — Yes (no workflow semantics)

## Scenario acceptance criteria

### Travel single-service budget

| Step | Expected behavior |
|------|------------------|
| Agent gets token with `budget.max_amount: 500` | Token issued with budget constraint |
| Agent searches flights (returns options $280-$550) | Success, read capability |
| Agent attempts to book $550 flight | Service checks budget, allows ($550 > budget check... depends on whether pre-check uses range_min or actual) |
| Agent attempts to book with token budget $200 | Service rejects with `budget_exceeded` before execution |

### Travel multi-service binding

| Step | Expected behavior |
|------|------------------|
| Agent searches flights | Returns results with `quote_id` |
| Agent attempts to book without quote_id | Rejected with `binding_missing` |
| Agent waits 20 minutes, attempts to book with stale quote | Rejected with `binding_stale` |
| Agent refreshes search, books with fresh quote | Success |
