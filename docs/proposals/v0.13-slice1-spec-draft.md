# ANIP v0.13 Slice 1: Spec Draft — Budget, Binding, and Control Requirements

## Overview

This document defines the concrete protocol additions for v0.13 Slice 1. It covers exact field names, validation rules, failure types, and service behavior requirements.

**Key design decisions:**
- Budget lives in one canonical location: `token.constraints.budget`
- Budget is a pre-execution MUST reject — no post-execution "blessed overspend"
- Binding is lightweight: field-presence and staleness checks only — no stateful resolution
- Permissions stays token-only — invoke-evaluable controls are checked at invoke time
- Invocation-request `budget` is a negotiation hint, never widens authority

---

## 1. Enforceable Budget Constraints

### Canonical location: `token.constraints.budget`

Budget is an **authorization constraint** carried in the delegation token's `constraints` block. It is not in `purpose_parameters`, not in scope strings, and not inferred from invocation hints.

**Token issuance request:**

```json
{
  "scope": ["travel.book"],
  "subject": "agent:trip-planner",
  "capability": "book_flight",
  "budget": {
    "currency": "USD",
    "max_amount": 500
  }
}
```

**Token claims (JWT):**

```json
{
  "constraints": {
    "max_delegation_depth": 3,
    "concurrent_branches": "allowed",
    "budget": {
      "currency": "USD",
      "max_amount": 500
    }
  }
}
```

### Precedence rule

| Source | Role | Authority |
|--------|------|-----------|
| `token.constraints.budget` | Enforceable authorization constraint | MUST enforce |
| Invocation request `budget` (§6.3) | Negotiation hint from the caller | MAY use for dynamic cost resolution; MUST NOT widen token budget |

If both are present, the token budget is the ceiling. The invocation budget hint can be lower but never higher.

### Budget enforcement rule

The service MUST enforce budget before executing the handler:

```
IF token.constraints.budget exists:
  LET budget_max = token.constraints.budget.max_amount
  LET budget_currency = token.constraints.budget.currency

  IF capability.cost.financial.currency != budget_currency:
    REJECT with budget_currency_mismatch

  IF capability.cost.certainty == "fixed" OR "estimated":
    LET check_amount = capability.cost.financial.range_min
      (or .typical if range_min is absent)
  ELSE IF capability.cost.certainty == "dynamic":
    LET check_amount = capability.cost.financial.upper_bound

  IF check_amount > budget_max:
    REJECT with budget_exceeded (BEFORE execution)

  EXECUTE handler

  # Post-execution: cost_actual is returned normally.
  # No "budget_warning" path — the pre-check is the enforcement.
```

**Why no post-execution warning:** Budget is a pre-execution control surface. If the service cannot determine cost before execution (no cost declaration, no upper_bound), budget enforcement does not apply — but the service SHOULD log a diagnostic. Services that declare cost MUST honor budget checks. This is what makes budget a real constraint, not an advisory.

### Budget narrowing in delegation chains

Budget constraints MUST narrow, never widen:

- If a parent token has `budget.max_amount = 500`, a child token MUST NOT have `budget.max_amount > 500`
- The service MUST reject token issuance that would widen a budget constraint
- If the parent has no budget, the child MAY introduce one

This matches the existing scope-narrowing rule for delegation.

### New failure types

**`budget_exceeded`:**

```json
{
  "success": false,
  "failure": {
    "type": "budget_exceeded",
    "detail": "Capability cost floor $200 exceeds delegated budget of $150",
    "resolution": {
      "action": "request_budget_increase",
      "requires": "delegation token with higher budget",
      "grantable_by": "human:manager@company.com"
    },
    "retry": false
  },
  "invocation_id": "inv-...",
  "budget_context": {
    "budget_max": 150,
    "budget_currency": "USD",
    "cost_check_amount": 200,
    "cost_certainty": "estimated"
  }
}
```

**`budget_currency_mismatch`:**

```json
{
  "success": false,
  "failure": {
    "type": "budget_currency_mismatch",
    "detail": "Token budget is in USD but capability cost is in EUR",
    "resolution": {
      "action": "request_matching_currency_delegation",
      "requires": "delegation token with EUR budget"
    },
    "retry": false
  }
}
```

## 2. Execution-Time Binding Requirements

### Design principle

Binding is **lightweight**: the service checks that a required field is present in the invocation parameters and optionally checks its age. The service does NOT validate that the binding "resolves" to a prior result — that is internal service logic, not a protocol requirement.

### Capability declaration: `requires_binding`

A capability MAY declare that it requires a bound reference from a prior step.

**Manifest declaration:**

```json
{
  "book_flight": {
    "description": "Book a flight reservation",
    "side_effect": { "type": "irreversible" },
    "requires_binding": [
      {
        "type": "quote",
        "field": "quote_id",
        "source_capability": "search_flights",
        "max_age": "PT15M"
      }
    ]
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | Yes | Binding type identifier (e.g., `quote`, `offer`, `price_lock`) |
| `field` | string | Yes | Which field in the invocation params must carry the reference |
| `source_capability` | string | No | Advisory — which capability produces this binding |
| `max_age` | string (ISO 8601 duration) | No | Maximum age before the binding is considered stale |

`source_capability` is informational only — the service does not validate it. It helps agents understand the expected workflow sequence.

### Invocation with binding

The caller MUST include the bound reference in the invocation parameters:

```json
{
  "parameters": {
    "flight_number": "AA100",
    "quote_id": "qt-a1b2c3d4"
  }
}
```

### Service enforcement

When a capability declares `requires_binding`:

1. **Missing binding**: If the required `field` is absent from parameters, the service MUST reject with `binding_missing`.
2. **Stale binding**: If `max_age` is declared and the binding was issued more than `max_age` ago, the service MUST reject with `binding_stale`. How the service determines binding age is an implementation detail (e.g., embedded timestamp in the binding value, server-side lookup).

There is no `binding_invalid` failure type at the protocol level. If the service's internal validation rejects a binding value, it returns a standard ANIP failure with an appropriate type (e.g., `invalid_parameters`).

### New failure types

**`binding_missing`:**

```json
{
  "success": false,
  "failure": {
    "type": "binding_missing",
    "detail": "Capability book_flight requires 'quote_id' (type: quote)",
    "resolution": {
      "action": "obtain_binding",
      "requires": "invoke search_flights to obtain a quote_id"
    },
    "retry": false
  }
}
```

**`binding_stale`:**

```json
{
  "success": false,
  "failure": {
    "type": "binding_stale",
    "detail": "Binding 'quote_id' has exceeded max_age of 15 minutes",
    "resolution": {
      "action": "refresh_binding",
      "requires": "invoke search_flights again for a fresh quote_id"
    },
    "retry": false
  }
}
```

## 3. Explicit Control Requirements

### Capability declaration: `control_requirements`

A capability MAY declare explicit control requirements. These are split into two categories based on when they can be evaluated:

**Token-evaluable** — checkable from the token alone, shown in `/anip/permissions`:
- `cost_ceiling` — token must carry a `constraints.budget`
- `stronger_delegation_required` — token must have explicit capability binding

**Invoke-evaluable** — checkable only at invocation time:
- `bound_reference` — a binding field must be present in params
- `freshness_window` — the binding must be recent enough

**Manifest declaration:**

```json
{
  "execute_trade": {
    "description": "Execute a market trade",
    "side_effect": { "type": "irreversible" },
    "cost": { "certainty": "dynamic", "financial": { "currency": "USD", "upper_bound": 10000 } },
    "control_requirements": [
      { "type": "cost_ceiling", "enforcement": "reject" },
      { "type": "bound_reference", "field": "market_quote_id", "enforcement": "reject" },
      { "type": "freshness_window", "max_age": "PT5M", "enforcement": "reject" }
    ]
  }
}
```

### Enforcement

| Value | Service behavior |
|-------|-----------------|
| `reject` | MUST reject if not satisfied |
| `warn` | SHOULD include a warning but MAY proceed |

### New failure type

**`control_requirement_unsatisfied`:**

```json
{
  "success": false,
  "failure": {
    "type": "control_requirement_unsatisfied",
    "detail": "Capability execute_trade requires cost_ceiling in delegation token",
    "resolution": {
      "action": "request_budget_bound_delegation",
      "requires": "delegation token with budget constraint"
    },
    "retry": false
  },
  "unsatisfied_requirements": ["cost_ceiling"]
}
```

## Permission discovery extension

`POST /anip/permissions` response includes control requirement status for **token-evaluable** requirements only:

```json
{
  "available": [
    {
      "capability": "search_flights",
      "scope_match": "travel.search"
    }
  ],
  "restricted": [
    {
      "capability": "execute_trade",
      "reason": "missing cost_ceiling in delegation",
      "grantable_by": "human:admin@company.com",
      "unmet_token_requirements": ["cost_ceiling"]
    }
  ]
}
```

`unmet_token_requirements` lists only token-evaluable controls that are not satisfied. Invoke-evaluable controls (`bound_reference`, `freshness_window`) are NOT included — they can only be checked at invoke time with actual parameters.

## Transport parity

All new fields are transport-neutral:

| Surface | Transport handling |
|---------|-------------------|
| `constraints.budget` in token | Encoded in JWT claims (all transports) |
| `requires_binding` in manifest | Part of capability declaration (all transports) |
| `control_requirements` in manifest | Part of capability declaration (all transports) |
| Binding fields in invoke | In the invocation request body/params (all transports) |
| New failure types | In the failure response body (all transports) |

---

## What this does NOT introduce

- No workflow ordering or sequencing
- No approval protocol
- No policy language
- No stateful binding resolution at the protocol level
- No post-execution budget override
- No generalized dependency graph

These are pure pre-execution control checks: deterministic, stateless at the protocol level, enforceable at the invoke boundary.

---

## Decision test

1. **Does it help the agent know more before execution?** — Yes (budget limits, binding requirements, control posture)
2. **Does it let the service reject earlier and more clearly?** — Yes (4 new deterministic rejection types)
3. **Does it remove repeated safety wrapper code?** — Yes (budget wrappers, binding validation, freshness checks)
4. **Would it still make sense with separate policy systems?** — Yes (execution-boundary checks, not policy)
5. **Does it preserve ANIP as execution/control protocol?** — Yes (no workflow semantics)

## Scenario acceptance criteria

### Travel single-service budget

| Step | Expected behavior | Rule |
|------|------------------|------|
| Token issued with `constraints.budget = {USD, 500}` | Token has enforceable budget | |
| Agent searches flights (returns $280-$550 options) | Success (read, no budget check) | |
| Agent books $280 flight with budget $500 token | Success (`range_min $200 < budget $500`) | Pre-check: `range_min <= budget_max` |
| Agent books with budget $150 token | Rejected: `budget_exceeded` | Pre-check: `range_min $200 > budget $150` |
| Child token issued with budget $600 from parent $500 | Rejected: budget narrowing violation | `child.budget_max <= parent.budget_max` |

### Travel multi-service binding

| Step | Expected behavior | Rule |
|------|------------------|------|
| Agent searches flights | Returns results. Service MAY include `quote_id` in result | |
| Agent books without `quote_id` | Rejected: `binding_missing` | `requires_binding.field` absent |
| Agent waits 20 min, books with old `quote_id` | Rejected: `binding_stale` (if `max_age: PT15M`) | Binding age > `max_age` |
| Agent refreshes search, books with fresh `quote_id` | Success | Binding present and fresh |
