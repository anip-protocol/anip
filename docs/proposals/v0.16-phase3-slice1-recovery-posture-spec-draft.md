# ANIP v0.16 Phase 3 Slice 1: Spec Draft â€” Recovery Posture

## Overview

This document defines a narrow first slice for Phase 3: make failure recovery posture more machine-readable by adding `resolution.recovery_class` to failure responses.

**Key design decisions:**
- One new field on `Resolution`: `recovery_class` â€” an advisory broad category for the recovery path
- Two new canonical `resolution.action` values: `retry_now`, `revalidate_state`
- `recovery_class` is deterministically derived from `resolution.action` via mandatory mapping rules â€” it is a convenience grouping, NOT an independent signal
- `recovery_class` does NOT change the meaning of `retry` â€” existing `retry: false` actions keep `retry: false`
- No new failure types
- No changes to the canonical action rule from v0.15 â€” services MUST use canonical actions

**Relationship to rejected `block_class`:**
v0.15 rejected `block_class` because it was an orthogonal taxonomy duplicating existing information. `recovery_class` differs:
1. Lives on `Resolution` (recovery guidance), not on the failure itself
2. Has mandatory actionâ†’class mapping â€” derivable from the action, not independent
3. Is about recovery strategy ("what kind of move"), not block severity

**What this does NOT change:**
- `retry` keeps its current meaning and values â€” no forced changes
- No new failure types
- No new actions beyond `retry_now` and `revalidate_state`
- No relaxation of v0.15's canonical-action rule
- No permission discovery changes

## 1. New Field: `resolution.recovery_class`

### Problem

`resolution.action` is canonical (v0.15), but agents still branch on the CATEGORY of recovery before choosing specific handling. Every agent maintains the same mapping:

```
if action in [request_broader_scope, request_budget_increase, request_new_delegation, ...]:
    # need a different token
elif action in [obtain_binding, refresh_binding]:
    # need to refresh a prerequisite
elif action in [wait_and_retry]:
    # wait for external state
...
```

That mapping is the same everywhere. It belongs in the protocol.

### Addition

Add `recovery_class` to the `Resolution` object:

```yaml
resolution:
  action: request_broader_scope
  recovery_class: redelegation_then_retry
  requires: "delegation.scope += travel.book"
  grantable_by: "human:alice@company.com"
```

### `recovery_class` vocabulary

| Value | Meaning |
|-------|---------|
| `retry_now` | Retry immediately, no prerequisite change needed |
| `wait_then_retry` | Wait for external availability, then retry |
| `refresh_then_retry` | Refresh a prerequisite or binding before retrying |
| `redelegation_then_retry` | Obtain a different token shape before retrying |
| `revalidate_then_retry` | Recheck external state or assumptions before retrying |
| `terminal` | No recovery path exists for this delegation chain |

### Advisory nature

`recovery_class` is an advisory grouping. It does NOT override or redefine `retry`:
- A failure with `recovery_class: "redelegation_then_retry"` and `retry: false` is valid â€” it means "a different token is needed, but the exact same request won't succeed even with retry"
- A failure with `recovery_class: "terminal"` and `retry: false` is the strongest signal â€” no path forward

The existing `retry` field keeps its current semantics exactly as defined today.

## 2. Mandatory Action â†’ Recovery Class Mapping

Services MUST emit `recovery_class` consistent with `resolution.action`. This makes `recovery_class` derivable from the action.

### Mapping table

| `resolution.action` | Required `recovery_class` |
|---------------------|--------------------------|
| `retry_now` | `retry_now` |
| `wait_and_retry` | `wait_then_retry` |
| `obtain_binding` | `refresh_then_retry` |
| `refresh_binding` | `refresh_then_retry` |
| `obtain_quote_first` | `refresh_then_retry` |
| `revalidate_state` | `revalidate_then_retry` |
| `request_broader_scope` | `redelegation_then_retry` |
| `request_budget_increase` | `redelegation_then_retry` |
| `request_budget_bound_delegation` | `redelegation_then_retry` |
| `request_matching_currency_delegation` | `redelegation_then_retry` |
| `request_new_delegation` | `redelegation_then_retry` |
| `request_capability_binding` | `redelegation_then_retry` |
| `request_deeper_delegation` | `redelegation_then_retry` |
| `escalate_to_root_principal` | `terminal` |

Note: `retry` values on these actions are UNCHANGED from their current spec definitions. `recovery_class` is layered on top without altering existing semantics.

## 3. New Canonical Actions

Two new canonical actions are added. No other action changes.

### `retry_now`

Use when the failure is transient and immediate retry is structurally reasonable â€” no wait, no prerequisite refresh, no redelegation.

```yaml
failure:
  type: internal_error
  detail: "temporary backend timeout"
  resolution:
    action: retry_now
    recovery_class: retry_now
  retry: true
```

This is distinct from `wait_and_retry`, which implies an external wait condition and optional `estimated_availability`.

### `revalidate_state`

Use when the action depends on external state that may have changed. Retrying without rechecking would be unsafe or wasteful.

```yaml
failure:
  type: capability_unavailable
  detail: "environment status changed since pre-check"
  resolution:
    action: revalidate_state
    recovery_class: revalidate_then_retry
    requires: "refresh environment status"
  retry: true
```

## 4. Examples with Recovery Class

### Scope failure

```yaml
failure:
  type: scope_insufficient
  detail: "delegation chain lacks scope: travel.book"
  resolution:
    action: request_broader_scope
    recovery_class: redelegation_then_retry
    requires: "delegation.scope += travel.book"
    grantable_by: "human:alice@company.com"
  retry: true
```

### Budget exceeded

```yaml
failure:
  type: budget_exceeded
  detail: "cost $420 exceeds budget $200"
  resolution:
    action: request_budget_increase
    recovery_class: redelegation_then_retry
    requires: "delegation token with higher budget"
    grantable_by: "human:alice@company.com"
  retry: false
```

Note: `retry: false` is preserved from the current spec â€” the same request with the same token won't succeed. `recovery_class: redelegation_then_retry` tells the agent the recovery path is getting a different token.

### Stale binding

```yaml
failure:
  type: binding_stale
  detail: "quote_id exceeded max_age PT15M"
  resolution:
    action: refresh_binding
    recovery_class: refresh_then_retry
    requires: "refresh quote_id via search_flights"
  retry: false
```

Note: `retry: false` preserved â€” same stale quote won't work. `recovery_class` tells agent to refresh the prerequisite.

### Temporary unavailability

```yaml
failure:
  type: capability_unavailable
  detail: "book_flight is temporarily unavailable"
  resolution:
    action: wait_and_retry
    recovery_class: wait_then_retry
    estimated_availability: "2026-04-01T10:15:00Z"
  retry: true
```

### Non-delegable action

```yaml
failure:
  type: non_delegable_action
  detail: "destroy_cluster requires direct principal action"
  resolution:
    action: escalate_to_root_principal
    recovery_class: terminal
  retry: false
```

### Transient error

```yaml
failure:
  type: internal_error
  detail: "temporary backend timeout"
  resolution:
    action: retry_now
    recovery_class: retry_now
  retry: true
```

## 5. What This Does NOT Introduce

- No new failure types (no `stale_assumption` â€” use existing types with `revalidate_state` action)
- No new action beyond `retry_now` and `revalidate_state`
- No changes to existing `retry` values
- No relaxation of canonical-action-only rule from v0.15
- No capability-level `refresh_via` or `follow_up_capabilities`
- No composition or sequencing hints
- No compensation semantics
- No permission discovery changes
- No workflow graphs or planners

## 7. Scenario Acceptance Criteria

### Travel: stale quote recovery

| Step | Expected |
|------|----------|
| Agent books with stale quote | `binding_stale` failure |
| `resolution.action` | `refresh_binding` |
| `recovery_class` | `refresh_then_retry` |
| `retry` | `false` (preserved â€” same stale quote won't work) |

### DevOps: temporary unavailability

| Step | Expected |
|------|----------|
| Agent invokes unavailable capability | `capability_unavailable` failure |
| `resolution.action` | `wait_and_retry` or `retry_now` |
| `recovery_class` | `wait_then_retry` or `retry_now` (consistent with action) |
| `retry` | `true` |

### Authority: missing scope

| Step | Expected |
|------|----------|
| Agent invokes without scope | `scope_insufficient` failure |
| `resolution.action` | `request_broader_scope` |
| `recovery_class` | `redelegation_then_retry` |
| `retry` | `true` |

### Terminal: non-delegable

| Step | Expected |
|------|----------|
| Agent invokes non-delegable action | `non_delegable_action` failure |
| `resolution.action` | `escalate_to_root_principal` |
| `recovery_class` | `terminal` |
| `retry` | `false` |
