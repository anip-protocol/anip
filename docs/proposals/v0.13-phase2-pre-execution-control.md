# ANIP Phase 2 Proposal: Pre-Execution Control Surfaces

## Why Phase 2 Exists

Phase 1 made identity and lineage explicit:

- `task_id`
- `parent_invocation_id`
- stronger audit queryability

That removed a meaningful class of observability glue.

But the worked examples still keep landing on `PARTIAL` for the same reason:

the interface can increasingly explain what happened, but it still does not
carry enough *enforceable pre-execution control* to stop teams from rebuilding
safety logic around the call boundary.

That is the Phase 2 problem.

## Core Goal

Phase 2 should reduce **safety glue** by making more pre-execution constraints:

- explicit
- machine-readable
- enforceable
- portable across runtimes

The key word is **control**, not workflow.

## What Phase 2 Mostly Is

Phase 2 should mostly be understood as:

- hardening existing ANIP surfaces
- making those surfaces more enforceable
- making them more portable across runtimes
- making them easier to validate through scenarios

ANIP already has important building blocks in this area:

- cost signaling
- permission discovery
- structured failure semantics
- blocked-action resolution hints

Phase 2 should not read like the introduction of a brand-new subsystem.

It should read like:

> complete and harden the existing pre-execution control surfaces so teams do
> not have to rebuild them as bespoke wrapper logic

## What Phase 2 Should Remove

Today teams still keep writing:

- budget wrappers
- approval guards
- "do not invoke unless..." prompt hacks
- external preflight policy checks
- blocked-action branching code
- manual escalation glue

Phase 2 should reduce that class of logic.

## What Phase 2 Is Not

Phase 2 should **not** turn ANIP into:

- a workflow engine
- a policy engine
- an approval system
- an orchestration DSL
- a full business-rules platform

ANIP should stay focused on the interface boundary:

> what must be knowable or enforceable before an agent acts

## The Core Phase 2 Thesis

ANIP should evolve from:

- "the interface can describe the action"

to:

- "the interface can express and enforce more of the conditions under which the action is allowed"

That is the next meaningful jump.

## Recommended Phase 2 Scope

Phase 2 should focus on five areas:

1. enforceable budget and cost constraints
2. stronger authority and permission signaling
3. explicit preconditions and control requirements
4. clearer blocked-action and escalation semantics
5. lightweight approval posture declarations

These are ordered by priority, not by strict implementation dependency.

## 1. Enforceable Budget And Cost Constraints

### Why

This is the clearest remaining gap in the current examples.

Right now ANIP can increasingly expose:

- estimated cost
- cost range
- cost posture

But the system still ends up `PARTIAL` when the agent must honor a hard budget,
because visibility alone is not enough.

### Goal

Make budget and cost limits enforceable before execution, not merely visible.

### Candidate Additions

- stronger semantics for turning existing cost signaling into enforceable control
- clearer delegation-scope conventions for budget constraints
- more explicit budget/cost binding semantics in token purpose or constraints
- stronger service-side requirement to reject invocation when declared cost
  exceeds bound constraints
- clearer distinction between:
  - known fixed cost
  - bounded estimate
  - unresolved dynamic cost

### Glue Removed

- budget wrapper code
- external "check budget first" policies
- manual cost-guard orchestration
- prompt-based "please stay under X" hacks

### Design Constraint

This should not require ANIP to own budget policy definition globally.

It only needs ANIP to make budget-relevant constraints expressible and
enforceable at the execution boundary.

## 2. Stronger Authority And Permission Signaling

### Why

Permission discovery already exists, which is a major strength.

But in practical systems teams still need better protocol-level signals around:

- what authority is missing
- whether it is grantable
- whether it is delegated
- whether a stronger credential is required
- whether the block is hard or potentially resolvable

### Goal

Make it easier for agents and services to reason about authority before
execution, and easier for implementors to avoid trial-and-error permission
wrappers.

### Candidate Additions

- richer restricted/denied capability metadata
- more consistent authority-related resolution actions
- explicit distinction between:
  - insufficient scope
  - insufficient role/classification
  - non-delegable authority
- clearer protocol guidance for "grantable" vs "non-grantable" blocks

### Glue Removed

- permission probing glue
- auth wrapper logic
- ad hoc grant-request routing
- hand-built "why did this fail?" translation layers

### Design Constraint

ANIP should describe authority posture and blocked-action semantics.

It should not become the organization’s IAM product.

## 3. Explicit Preconditions And Control Requirements

### Why

Some scenarios stay `PARTIAL` not because the system lacks visibility, but
because it lacks a machine-readable way to say:

- this action is only valid if these conditions are satisfied first
- this action is only valid when bound to a prior quote, offer, or check
- this control requirement must hold at the exact execution boundary

This is especially visible in multi-service scenarios, where planning output
from one service needs to become an enforceable control on another.

### Goal

Make preconditions and control requirements first-class enough that services can
enforce them before side effects happen, without defining workflows.

### Candidate Additions

- explicit precondition or control-requirement declarations on capabilities
- machine-readable requirement types such as:
  - bound offer or quote reference
  - cost ceiling
  - stronger delegation requirement
  - freshness or validity window
- clearer distinction between:
  - informational prerequisite
  - enforceable control requirement
- service-side guidance for rejecting invocation when required control inputs
  are missing or invalid

### Glue Removed

- cross-step wrapper logic
- ad hoc "only invoke if ..." guards
- custom binding between planning output and execution input
- service-to-service validation glue

### Design Constraint

This is not workflow sequencing.

It is a way to say:

> these conditions must hold before this action is allowed to execute

## 4. Clearer Blocked-Action And Escalation Semantics

### Why

Even when an action is blocked correctly, teams still build custom logic to
decide:

- should the agent retry
- should it wait
- should it ask a human
- should it request a broader delegation
- should it replan

That is orchestration glue created by weak blocked-action signaling.

### Goal

Make blocked actions more predictable and more actionable without introducing
workflow definitions.

### Candidate Additions

- tighter vocabulary for `resolution.action`
- stronger guidance around:
  - `retry`
  - `retry_after_change`
  - `request_scope_grant`
  - `escalate`
  - `wait`
  - `replan`
  - `compensate`
- clearer service guidance for when a block is:
  - terminal
  - conditionally recoverable
  - externally resolvable

### Glue Removed

- bespoke retry trees
- custom escalation branches
- wrapper logic for interpreting blocked actions
- manual "if this kind of failure, then do X" orchestration

### Design Constraint

This is still about structured recovery semantics, not workflow programming.

## 5. Lightweight Approval Posture Declarations

### Why

Some of the most painful glue comes from actions that are technically possible
but should not execute without explicit approval.

Today that is often hidden outside the interface and reconstructed in:

- wrappers
- orchestration code
- custom portals
- policy middleware

### Goal

Let services declare approval posture in a deterministic way without ANIP
becoming the approval system itself.

### Candidate Additions

- deterministic approval posture fields such as:
  - `approval_requirement: none | external | delegated | non_delegable`
  - `approval_authority_class: <class>`
  - `approval_path: external | stronger_delegation | human_handoff`
- clearer blocked-action resolution when approval is the only path forward
- optional linkage fields for approval evidence later, if justified

### Glue Removed

- invisible approval guard logic
- ad hoc "ask a human first" branches
- duplicated approval-state interpretation

### Design Constraint

Approval-aware declarations should be Phase 2 territory.

Approval workflows themselves should remain outside ANIP.

This is why this area is listed fifth, not first.

## Success By Slice

Phase 2 should be validated slice by slice against the scenario set.

### Slice 1 Success

Slice 1 should focus on:

- enforceable budget and cost constraints
- explicit control requirements where needed for execution-time enforcement

It succeeds when:

- the single-service travel budget scenario moves from `PARTIAL` toward
  `HANDLED`
- the multi-service travel budget scenario moves from `PARTIAL` toward
  `HANDLED`
- the remaining glue is no longer basic budget wrapper logic, but only
  organization-specific policy choices

### Slice 2 Success

Slice 2 should focus on:

- stronger authority signaling
- clearer blocked-action and escalation semantics

It succeeds when:

- authority-gap scenarios move toward `HANDLED`
- implementors no longer need custom auth-wrapper logic to explain whether a
  block is grantable, non-delegable, or escalatable
- the remaining glue is mostly organizational routing, not protocol gap

### Slice 3 Success

Slice 3 should focus on:

- lightweight approval posture declarations

It succeeds when:

- approval-related scenarios reduce bespoke approval-guard glue
- the evaluator can classify approval posture deterministically
- ANIP still does not own the approval workflow itself

## What Phase 2 Should Probably Not Do Yet

The following ideas are tempting but should be treated as out of scope for the
first Phase 2 slice:

- approval request protocol
- policy language embedded in ANIP
- generalized workflow state machines
- long-running step orchestration
- execution DAG definitions
- built-in business-rule evaluation engine

Those are all plausible future adjacent systems.

They are not the right next ANIP move.

## Good Phase 2 Decision Test

A Phase 2 addition is likely healthy if it answers "yes" to most of these:

1. Does it help the agent know more before execution?
2. Does it let the service reject or shape action earlier and more clearly?
3. Does it remove repeated safety wrapper code?
4. Would it still make sense if the team uses separate policy or workflow systems?
5. Does it preserve ANIP as an execution/control protocol rather than a workflow runtime?

If the answer is mostly no, it is probably scope creep.

## Recommended Sequencing Inside Phase 2

### Slice 1: Budget, Binding, And Constraint Enforcement

Focus on:

- hardening the cost/budget contract
- making enforceable constraints clearer
- improving spec language around constraint rejection

This is the highest-value next step because it is the clearest current gap in
the examples.

### Slice 2: Authority And Blocked-Action Clarity

Focus on:

- richer permission/authority outcomes
- stronger blocked-action semantics
- more standardized recovery vocabulary

This should reduce the next-largest class of safety glue.

### Slice 3: Approval-Aware Interface Hints

Focus on:

- lightweight approval declarations
- clearer approval-related blocked-action signaling

This should only happen once slices 1 and 2 are clear.

## What Success Looks Like

Phase 2 is successful if the next round of scenario evaluation starts shifting:

- from `PARTIAL` because the system only *knows*
- toward `HANDLED` because the system can now *enforce or block correctly before execution*

That is the right outcome to optimize for.

## The Relationship To Scenario Validation

Phase 2 should be judged through `Execution Scenario Validation`, not just spec
elegance.

That means the real questions are:

- does this remove budget wrapper code
- does this reduce approval guard glue
- does this reduce permission probing
- does this reduce blocked-action orchestration

If a proposed Phase 2 feature does not improve those results in scenario runs,
it is probably not the right feature.

## The Strongest Short Version

> Phase 1 helped teams understand what happened. Phase 2 should help systems make better decisions before execution happens.

And the developer-pain version:

> If teams are still writing budget checks, approval guards, and blocked-action wrappers outside the interface, Phase 2 should focus on those gaps before ANIP tries to do anything more ambitious.
