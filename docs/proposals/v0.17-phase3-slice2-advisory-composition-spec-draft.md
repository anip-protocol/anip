# ANIP v0.17 Phase 3 Slice 2: Spec Draft — Advisory Composition Hints

## Overview

This document defines concrete additions for v0.17 Phase 3 Slice 2: advisory composition hints on capability declarations.

**Key design decisions:**
- Two new optional fields on `CapabilityDeclaration`: `refresh_via`, `verify_via`
- Both are simple string arrays referencing capability names within the same service manifest (same-manifest only, no cross-service references)
- Both are advisory — not executable, not mandatory ordering, not enforcement surfaces
- `follows` is NOT added — it duplicates the inverse of existing `requires` without a consistency rule
- No new failure types, no new resolution actions, no recovery_class changes

**What this does NOT add:**
- `follows` — forward relationships are derivable from `requires` (which is the canonical prerequisite graph). Adding an inverse creates drift risk with no sync rule.

**What already exists (unchanged):**

| Existing field | Location | Purpose |
|----------------|----------|---------|
| `requires` | CapabilityDeclaration | Navigational prerequisite relationships — "search before book" (advisory, not enforced) |
| `composes_with` | CapabilityDeclaration | Loose composition — "works well with these capabilities" |
| `requires_binding.source_capability` | BindingRequirement | Which capability produces the required binding (advisory) |
| `cost.determined_by` | Cost | Which capability resolves the actual cost (advisory) |

The new fields fill gaps that existing fields don't cover:
- **refresh_via**: which capability refreshes stale state (recovery guidance — complements `recovery_class: refresh_then_retry`)
- **verify_via**: which capability confirms side effects succeeded (post-action verification, not pre-action prerequisites)

---

## 1. New Fields on CapabilityDeclaration

### `refresh_via`

Which capabilities can refresh the state or prerequisites this capability depends on. Complements the recovery posture from v0.16: when `recovery_class` is `refresh_then_retry`, `refresh_via` tells the agent WHERE to refresh.

**All referenced capabilities MUST exist in the same service manifest.** Cross-service refresh relationships are outside the protocol boundary.

```json
{
  "book_flight": {
    "description": "Book a flight reservation",
    "refresh_via": ["search_flights"],
    "requires_binding": [{"type": "quote", "field": "quote_id", "source_capability": "search_flights", "max_age": "PT15M"}]
  }
}
```

When `book_flight` fails with `binding_stale` and `recovery_class: "refresh_then_retry"`, the agent reads `refresh_via: ["search_flights"]` and knows to re-invoke `search_flights`. Without this field, the agent infers it from `requires_binding.source_capability` — which works for bindings but not for other stale-state scenarios (e.g., inventory changes, environment status drift).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `refresh_via` | `string[]` | No | Capability names (same manifest) that can refresh prerequisites or stale state |

### `verify_via`

Which capabilities can confirm that this capability's side effects succeeded. Useful after irreversible or high-risk actions where the agent should verify the outcome.

**All referenced capabilities MUST exist in the same service manifest.**

```json
{
  "deploy_service": {
    "description": "Deploy a new service version",
    "side_effect": {"type": "irreversible"},
    "verify_via": ["get_deployment_status"]
  }
}
```

After invoking `deploy_service`, an agent reads `verify_via: ["get_deployment_status"]` and knows to check deployment health. Without this field, the agent relies on docs, prompts, or hard-coded verification maps.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `verify_via` | `string[]` | No | Capability names (same manifest) that can verify this capability's side effects |

---

## 2. Advisory Nature

Both fields are strictly advisory:

- They do NOT create execution requirements
- They do NOT replace `requires_binding` (which is enforceable)
- They do NOT replace `control_requirements` (which are enforceable)
- They do NOT imply mandatory ordering
- They are hints that reduce guesswork — nothing more

A service MAY include them. A service MAY omit them. An agent MAY ignore them. An agent that uses them gets less glue; an agent that ignores them gets the same protocol behavior as before.

---

## 3. Same-Manifest Scope

Both `refresh_via` and `verify_via` reference capability names within the same service manifest. They are NOT cross-service references.

This is intentional:
- A service knows its own capability relationships
- Cross-service composition is an orchestration concern, not a protocol concern
- A service-qualified reference shape (e.g., `"service:travel/search_flights"`) would require service discovery and registration — far beyond this slice's scope

If a capability's refresh or verification path is on a different service, the service should NOT declare it in these fields. That relationship lives in the orchestration layer.

---

## 4. Why Not `follows`?

The proposal considered a `follows` field (common next steps after this capability). It was dropped because:

1. `follows` is the inverse of `requires` — if `book_flight.requires = [search_flights]`, then `search_flights.follows = [book_flight]` is derivable
2. Having both creates drift risk: `search_flights.follows` could say `[book_flight]` while `book_flight.requires` omits `search_flights`, with no precedence rule
3. Agents can derive forward relationships by reading `requires` across all capabilities in the manifest — this is a one-time manifest read, not a per-invocation cost

`requires` is the canonical prerequisite graph. Adding an inverse without a sync rule adds concept weight without proportional value.

---

## 5. Manifest Surface

These fields appear in the manifest alongside existing capability declarations:

```json
{
  "capabilities": {
    "search_flights": {
      "description": "Search available flights",
      "requires": []
    },
    "book_flight": {
      "description": "Book a flight reservation",
      "requires": [{"capability": "search_flights", "reason": "must select from available flights"}],
      "requires_binding": [{"type": "quote", "field": "quote_id", "source_capability": "search_flights", "max_age": "PT15M"}],
      "refresh_via": ["search_flights"]
    },
    "deploy_service": {
      "description": "Deploy a new service version",
      "side_effect": {"type": "irreversible"},
      "verify_via": ["get_deployment_status"]
    }
  }
}
```

---

## 6. What This Does NOT Introduce

- No `follows` field (derivable from `requires`)
- No execution ordering or sequencing
- No conditional transitions
- No workflow graphs or DAGs
- No cross-service references
- No new failure types
- No new resolution actions
- No recovery_class changes
- No permission discovery changes
- No changes to existing fields
- No compensation semantics

---

## 7. Scenario Acceptance Criteria

### Travel: stale quote refresh discovery

| Step | Expected |
|------|----------|
| Agent reads `book_flight` manifest | Sees `refresh_via: ["search_flights"]` |
| Book attempt fails with `binding_stale` | `recovery_class: "refresh_then_retry"` |
| Agent reads `refresh_via` | Calls `search_flights` to get fresh quote — no guessing |

### DevOps: post-deployment verification

| Step | Expected |
|------|----------|
| Agent reads `deploy_service` manifest | Sees `verify_via: ["get_deployment_status"]` |
| Agent deploys successfully | Calls `get_deployment_status` to verify — no wrapper logic |

### Finance: post-trade verification

| Step | Expected |
|------|----------|
| Agent reads `execute_trade` manifest | Sees `verify_via: ["get_trade_status"]` |
| Agent executes trade | Calls `get_trade_status` to confirm execution |
