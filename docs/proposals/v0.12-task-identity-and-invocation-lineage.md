# ANIP v0.12: Task Identity and Invocation Lineage

## Motivation

Teams building agent systems repeatedly construct custom correlation and trace-stitching logic because the ANIP interface does not carry enough identity to connect related invocations. This is one of the most common forms of "workflow glue" — code that exists not because the business domain is complex, but because the interface tells the agent too little.

Two specific gaps drive most of this glue:

1. **Task identity** — which task, workflow, or agent objective does this invocation belong to?
2. **Invocation lineage** — which prior invocation triggered this one?

Without protocol-level answers to these questions, teams build ad hoc correlation through custom metadata, logs, workflow engines, and header conventions. That glue is expensive, fragile, and repeated across every integration.

## Design constraint

ANIP should reduce workflow glue by making execution context explicit. ANIP should **not** evolve into a workflow engine, orchestration runtime, or state machine language.

The additions in this proposal are pure information-carrying fields. They do not introduce execution semantics, sequencing logic, or state management. They help agents, operators, and audit systems understand *what happened and why* — not *what should happen next*.

## Proposal

### 1. `task_id`

A caller-supplied identifier that groups related invocations under a single task, workflow, or agent objective.

**Where it appears:**

| Surface | Field | Notes |
|---------|-------|-------|
| Invocation request | `task_id` | Optional, caller-supplied, max 256 chars |
| Invocation response | `task_id` | Echoed if provided |
| Delegation token (purpose_parameters) | `task_id` | Carried forward from delegation |
| Audit entry | `task_id` | Recorded if present |

**Semantics:**

- `task_id` is opaque to the service — it does not interpret, validate, or enforce it
- It is the caller's responsibility to assign and propagate task identity
- Multiple invocations with the same `task_id` are understood to be part of the same logical task
- Services MUST record `task_id` in audit entries when present
- Services MUST echo `task_id` in invocation responses when provided in the request

**Invocation request example:**

```json
{
  "parameters": { "origin": "SEA", "destination": "SFO" },
  "client_reference_id": "step-3-search",
  "task_id": "trip-planning-2026-q2",
  "parent_invocation_id": "inv_a1b2c3"
}
```

**Relationship to `client_reference_id`:**

| Field | Scope | Purpose |
|-------|-------|---------|
| `task_id` | Groups multiple invocations | "All of these belong to the same task" |
| `client_reference_id` | Identifies a single invocation | "This specific call is step 3 of the search phase" |

Both are caller-supplied and opaque. `task_id` operates at a higher level of grouping. A single task may produce many invocations, each with its own `client_reference_id`.

**Delegation token integration:**

When issuing a delegation token, `task_id` can be included in `purpose_parameters`:

```json
{
  "scope": ["travel.search", "travel.book"],
  "subject": "agent:trip-planner",
  "purpose_parameters": {
    "task_id": "trip-planning-2026-q2",
    "budget_usd": 500
  }
}
```

The service MAY surface `task_id` from the delegation token in audit entries, enabling correlation even when the invoking agent does not explicitly include `task_id` in every request.

### 2. `parent_invocation_id`

A reference to the invocation that caused this invocation to happen.

**Where it appears:**

| Surface | Field | Notes |
|---------|-------|-------|
| Invocation request | `parent_invocation_id` | Optional, caller-supplied |
| Invocation response | `parent_invocation_id` | Echoed if provided |
| Audit entry | `parent_invocation_id` | Recorded if present |

**Semantics:**

- `parent_invocation_id` MUST be a valid `invocation_id` from a prior invocation (format: `inv_{hex12}`)
- It is the caller's responsibility to propagate parent identity
- The service does not validate that the referenced invocation exists — it records it as provided
- A chain of `parent_invocation_id` references forms an invocation tree

**Example: multi-step agent workflow**

```
Agent receives task: "book the cheapest SEA→SFO flight"

1. invoke search_flights
   → invocation_id: inv_a1b2c3
   → task_id: trip-2026

2. invoke check_availability (for cheapest result DL310)
   → invocation_id: inv_d4e5f6
   → task_id: trip-2026
   → parent_invocation_id: inv_a1b2c3

3. invoke book_flight (DL310)
   → invocation_id: inv_g7h8i9
   → task_id: trip-2026
   → parent_invocation_id: inv_d4e5f6
```

**Audit query:**

With these fields recorded, an operator can:

- Query by `task_id` to see all invocations for the trip-planning task
- Follow `parent_invocation_id` links to reconstruct the decision chain
- Understand that the booking happened *because* availability was confirmed *because* search returned that flight

This replaces custom trace-stitching code that teams currently build in orchestration layers.

## Audit entry shape (v0.12)

```json
{
  "invocation_id": "inv_g7h8i9",
  "capability": "book_flight",
  "actor_key": "agent:trip-planner",
  "root_principal": "human:samir@example.com",
  "event_class": "high_risk_success",
  "success": true,
  "task_id": "trip-2026",
  "client_reference_id": "step-3-book",
  "parent_invocation_id": "inv_d4e5f6",
  "timestamp": "2026-04-01T10:30:00Z"
}
```

## Transport parity

Both fields are transport-neutral:

| Transport | How `task_id` and `parent_invocation_id` are carried |
|-----------|------------------------------------------------------|
| HTTP | In the invocation request JSON body |
| stdio | In the `params` object of the JSON-RPC `anip.invoke` call |
| gRPC | In the `InvokeRequest` protobuf message |

No transport-specific handling is needed — these are part of the invocation request payload.

## What this does NOT introduce

- No execution ordering or sequencing semantics
- No requirement to validate parent invocation existence
- No workflow state management
- No approval or gate logic
- No changes to permission or delegation semantics

These are pure metadata fields that improve traceability without expanding protocol scope.

## Decision test

Applying the workflow-reduction guardrail questions:

1. **Does this make pre-execution reasoning clearer?** — Partially (lineage context helps agents understand their own history)
2. **Does it remove repeated custom wrapper/orchestration code?** — Yes (task correlation, trace stitching)
3. **Is it an interface semantic rather than a workflow definition?** — Yes (metadata, not execution logic)
4. **Would it still make sense if the team uses a separate orchestrator?** — Yes (the orchestrator benefits from these fields)
5. **Does it preserve ANIP's identity as an execution/control protocol?** — Yes

All five: yes. This is safe ANIP evolution.

## Future extensions (not in this proposal)

These are explicitly deferred:

- `root_invocation_id` — automatic root-of-chain tracking (adds complexity, can be derived from parent chain)
- Approval-aware declarations — Tier 2
- Composition/prerequisite hints — needs more design
- Recovery action vocabulary standardization — separate proposal
- Cross-service lineage query endpoint — Tier 3
