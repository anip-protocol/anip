# ANIP v0.12: Task Identity and Invocation Lineage

## Motivation

Teams building agent systems repeatedly construct custom correlation and trace-stitching logic because the ANIP interface does not carry enough identity to connect related invocations. This is one of the most common forms of "workflow glue" — code that exists not because the business domain is complex, but because the interface tells the agent too little.

Two specific gaps drive most of this glue:

1. **Task identity in invocations** — which task does this invocation belong to? ANIP already has `task_id` in the delegation `Purpose` model, but it does not surface in invocation requests, responses, or audit queries.
2. **Invocation lineage** — which prior invocation triggered this one? There is no protocol-level parent reference today.

Without these fields flowing through invocations and audit, teams build ad hoc correlation through custom metadata, logs, workflow engines, and header conventions.

## Design constraint

ANIP should reduce workflow glue by making execution context explicit. ANIP should **not** evolve into a workflow engine, orchestration runtime, or state machine language.

The additions in this proposal are pure information-carrying fields. They do not introduce execution semantics, sequencing logic, or state management.

## Proposal

### 1. Surface `task_id` in invocations and audit

ANIP already defines `task_id` as a first-class field in the delegation `Purpose` model (SPEC §4.3, `anip_core.models.Purpose`). This proposal extends its reach to the invocation surface and audit queries.

**Current state:**

- `task_id` exists in `Purpose` (delegation token)
- `task_id` is NOT in invocation requests or responses
- `task_id` is NOT queryable in audit

**Proposed changes:**

| Surface | Change |
|---------|--------|
| Invocation request | Add optional `task_id` field (max 256 chars) |
| Invocation response | Echo `task_id` if provided in request |
| Audit entry | Record `task_id` — sourced with the following precedence |
| Audit query | Add `task_id` as a query filter parameter |

**Precedence rule for `task_id`:**

When recording `task_id` in audit, the service resolves it as follows:

1. **If the delegation token has `purpose.task_id`** — the token's task identity is authoritative. If the invocation request also provides `task_id`, it MUST match the token value. A mismatch MUST be rejected with a `purpose_mismatch` failure. If the request omits `task_id`, the token's value is used.
2. **If the delegation token has no `purpose.task_id`** — the invocation request MAY provide `task_id`, and that value is recorded.
3. **If neither source provides a `task_id`** — the audit entry has no `task_id`.

This preserves the delegation model's purpose-binding semantics: a token issued for a specific task cannot be used to attribute invocations to a different task. Only tokens without task identity accept invocation-level task assignment.

**Invocation request example:**

```json
{
  "parameters": { "origin": "SEA", "destination": "SFO" },
  "client_reference_id": "step-3-search",
  "task_id": "trip-planning-2026-q2",
  "parent_invocation_id": "inv-a1b2c3d4e5f6"
}
```

**Relationship to existing fields:**

| Field | Source | Scope | Purpose |
|-------|--------|-------|---------|
| `purpose.task_id` | Delegation token | Token lifetime | "This token was issued for this task" |
| `task_id` (invocation) | Invoke request | Single invocation | "This specific call belongs to this task" (overrides token) |
| `client_reference_id` | Invoke request | Single invocation | "Caller's own correlation ID for this call" |

**Audit query extension:**

Add `task_id` as an optional query parameter to `POST /anip/audit`:

```bash
curl -X POST "https://service.example/anip/audit?task_id=trip-planning-2026-q2" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" -d '{}'
```

This returns all audit entries for the given task, replacing the custom client-side filtering teams currently build.

### 2. `parent_invocation_id`

A reference to the invocation that caused this invocation to happen.

**Where it appears:**

| Surface | Field | Notes |
|---------|-------|-------|
| Invocation request | `parent_invocation_id` | Optional, caller-supplied |
| Invocation response | `parent_invocation_id` | Echoed if provided |
| Audit entry | `parent_invocation_id` | Recorded if present |
| Audit query | `parent_invocation_id` | Filter parameter |

**Semantics:**

- `parent_invocation_id` SHOULD match the `inv-{hex12}` format of a prior `invocation_id`
- The service does **not** validate that the referenced invocation exists — it records the value as provided
- It is the caller's responsibility to propagate parent identity
- A chain of `parent_invocation_id` references forms an invocation tree

The field is syntactically validated (must look like an invocation ID) but not referentially validated (the service does not check that the parent invocation actually occurred). This keeps the protocol simple while still enabling lineage reconstruction.

**Example: multi-step agent workflow**

```
Agent receives task: "book the cheapest SEA→SFO flight"

1. invoke search_flights
   → invocation_id: inv-a1b2c3d4e5f6
   → task_id: trip-2026

2. invoke check_availability (for cheapest result DL310)
   → invocation_id: inv-d4e5f6a7b8c9
   → task_id: trip-2026
   → parent_invocation_id: inv-a1b2c3d4e5f6

3. invoke book_flight (DL310)
   → invocation_id: inv-c7d8e9f0a1b2
   → task_id: trip-2026
   → parent_invocation_id: inv-d4e5f6a7b8c9
```

**Audit query extension:**

Add `parent_invocation_id` as an optional query parameter:

```bash
# Find all invocations triggered by a specific parent
curl -X POST "https://service.example/anip/audit?parent_invocation_id=inv-a1b2c3d4e5f6" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" -d '{}'
```

This replaces custom trace-stitching code. An operator can now follow invocation chains through the audit API itself, not in a separate correlation system.

## Audit entry shape (v0.12)

```json
{
  "invocation_id": "inv-c7d8e9f0a1b2",
  "capability": "book_flight",
  "actor_key": "agent:trip-planner",
  "root_principal": "human:samir@example.com",
  "event_class": "high_risk_success",
  "success": true,
  "task_id": "trip-2026",
  "client_reference_id": "step-3-book",
  "parent_invocation_id": "inv-d4e5f6a7b8c9",
  "timestamp": "2026-04-01T10:30:00Z"
}
```

## Audit query parameters (v0.12)

Updated list of audit query parameters:

| Parameter | Type | Description | New in v0.12 |
|-----------|------|-------------|--------------|
| `capability` | string | Filter by capability name | No |
| `since` | string | ISO 8601 timestamp | No |
| `invocation_id` | string | Filter by specific invocation | No |
| `client_reference_id` | string | Filter by caller correlation ID | No |
| `task_id` | string | Filter by task identity | **Yes** |
| `parent_invocation_id` | string | Filter by parent invocation | **Yes** |
| `limit` | integer | Max entries to return | No |

## Transport parity

Both fields are transport-neutral:

| Transport | How `task_id` and `parent_invocation_id` are carried |
|-----------|------------------------------------------------------|
| HTTP | In the invocation request JSON body |
| stdio | In the `params` object of the JSON-RPC `anip.invoke` call |
| gRPC | In the `InvokeRequest` protobuf message |

No transport-specific handling is needed — these are part of the invocation request payload.

## What this does NOT introduce

- No execution ordering or sequencing semantics
- No referential validation of parent invocation existence
- No workflow state management
- No approval or gate logic
- No changes to permission or delegation semantics

These are pure metadata fields with audit query support that improve traceability without expanding protocol scope.

## Decision test

Applying the workflow-reduction guardrail questions:

1. **Does this make pre-execution reasoning clearer?** — Partially (lineage context helps agents understand their own history)
2. **Does it remove repeated custom wrapper/orchestration code?** — Yes (task correlation, trace stitching, audit filtering)
3. **Is it an interface semantic rather than a workflow definition?** — Yes (metadata + query, not execution logic)
4. **Would it still make sense if the team uses a separate orchestrator?** — Yes (the orchestrator benefits from these fields)
5. **Does it preserve ANIP's identity as an execution/control protocol?** — Yes

All five: yes. This is safe ANIP evolution.

## Future extensions (not in this proposal)

These are explicitly deferred:

- `root_invocation_id` — automatic root-of-chain tracking (adds complexity, can be derived from parent chain)
- Approval-aware declarations — Tier 2
- Composition/prerequisite hints — needs more design
- Recovery action vocabulary standardization — separate proposal
- Cross-service lineage query endpoint — Tier 3
