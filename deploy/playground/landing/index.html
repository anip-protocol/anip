Try these scenarios

Fully specified, copy-paste scenarios against the live travel showcase. Each shows what ANIP makes possible that REST, GraphQL, and MCP cannot.

Demo data: Available routes: SEA→SFO (3 flights: AA100 $420, UA205 $380, DL310 $280), SEA→LAX (2 flights), LAX→SEA, SEA→JFK. API key: demo-human-key

1. Search, evaluate, then book a flight

Goal: Find a flight from SEA to SFO, inspect its cost and side effects before booking, then book it with full context.
REST / GraphQL / MCP

Call the booking endpoint with a flight number. Discover the cost after being charged. No way to know this is irreversible until it's done. No budget check exists in the interface.

ANIP

Agent reads the manifest: book_flight is irreversible, costs $200–$800. Searches first, picks a flight, checks permissions, then books with full context. Every step is informed.

Step 1: Inspect the manifest — see that search_flights is read (safe) and book_flight is irreversible (permanent) 
curl -s https://travel.playground.anip.dev/anip/manifest | jq '.capabilities | to_entries[] | {name: .key, side_effect: .value.side_effect.type}'
Step 2: Get a delegation token with both search and booking scope 
curl -s -X POST https://travel.playground.anip.dev/anip/tokens \
  -H "Authorization: Bearer demo-human-key" \
  -H "Content-Type: application/json" \
  -d '{"scope": ["travel.search", "travel.book"], "subject": "agent:demo"}'
Save the token value from the response for the next steps.
Step 3: Search flights from SEA to SFO — this is a read operation, safe to call freely 
curl -s -X POST https://travel.playground.anip.dev/anip/invoke/search_flights \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"parameters": {"origin": "SEA", "destination": "SFO"}}'
You'll see 3 flights: AA100 ($420), UA205 ($380), DL310 ($280).
Step 4: Check permissions before attempting the irreversible booking 
curl -s -X POST https://travel.playground.anip.dev/anip/permissions \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" -d '{}'
Confirm book_flight is in the available bucket.
Step 5: Book the cheapest flight (DL310, $280) — the response includes cost_actual and invocation_id 
curl -s -X POST https://travel.playground.anip.dev/anip/invoke/book_flight \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"parameters": {"flight_number": "DL310"}}'
What ANIP adds: The agent knew the cost range, side-effect type, and its authority before acting. It searched first (safe read), evaluated options, checked permissions, then booked with full context. REST/GraphQL/MCP give none of this — the agent discovers constraints by failing.

2. Handle a permission block with recovery guidance

Goal: Get a token that can search but not book. Try to understand what's blocked and how to fix it — without attempting the operation.
REST / GraphQL / MCP

Agent calls the booking endpoint, gets 403 Forbidden. No information about why, who can fix it, or what's needed. Agent retries or gives up.

ANIP

Agent checks permissions before acting. The response tells it exactly what's restricted and who can grant the missing scope. The agent reports to its user instead of failing blindly.

Step 1: Get a token with only search scope (no booking permission) 
curl -s -X POST https://travel.playground.anip.dev/anip/tokens \
  -H "Authorization: Bearer demo-human-key" \
  -H "Content-Type: application/json" \
  -d '{"scope": ["travel.search"], "subject": "agent:demo"}'
Step 2: Check permissions — see what's available vs restricted vs denied 
curl -s -X POST https://travel.playground.anip.dev/anip/permissions \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" -d '{}'
Expected response shape: 
{
  "available": [
    { "capability": "search_flights", "scope_match": "travel.search" }
  ],
  "restricted": [
    { "capability": "book_flight", "reason": "missing scope: travel.book",
      "grantable_by": "human:samir@example.com" }
  ],
  "denied": []
}
Step 3: The agent now knows: "I can search flights, but booking requires travel.book scope. I should ask human:samir@example.com for that scope."
What ANIP adds: Structured recovery guidance. The agent doesn't just know it's blocked — it knows what scope is missing, who can grant it, and can tell the user exactly what to do. No 403-and-guess loop.

3. Audit the full action trail

Goal: After running Scenario 1 (search + book), review what happened — who acted, what was invoked, whether it succeeded, and verify the evidence is tamper-proof.
REST / GraphQL / MCP

No standard audit API. Check application logs manually (if they exist). No event classification, no lineage, no protocol-level retention policy.

ANIP

Agent queries /anip/audit to see every invocation with capability name, caller identity, event classification (low_risk_success for reads, high_risk_success for bookings), and timestamps. Merkle checkpoints prove the log hasn't been tampered with.

Prerequisite: Run Scenario 1 first (search flights, then book DL310) so there are audit entries to review.
Step 1: Query the audit log — you'll see your search (low_risk_success) and booking (high_risk_success) entries 
curl -s -X POST https://travel.playground.anip.dev/anip/audit \
  -H "Authorization: Bearer demo-human-key" \
  -H "Content-Type: application/json" -d '{}'
Step 2: Filter by capability to find just the booking 
curl -s -X POST "https://travel.playground.anip.dev/anip/audit?capability=book_flight" \
  -H "Authorization: Bearer demo-human-key" \
  -H "Content-Type: application/json" -d '{}'
Step 3: Check Merkle checkpoints — cryptographic proof that the audit log hasn't been modified 
curl -s https://travel.playground.anip.dev/anip/checkpoints
Each checkpoint includes a merkle_root and signature. If any audit entry were changed, the root would no longer match.
What ANIP adds: Protocol-level audit with event classification, delegation chain identity, and tamper-evident Merkle checkpoints. Every action is recorded, classified, and verifiable — not as an afterthought, but as part of the interface contract.
